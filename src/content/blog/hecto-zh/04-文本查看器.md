## 表示文本

操控 *文本* 在一个程序中正确学习是一个棘手的话题，正如你在本章中将看到的那样。暂且假设 *文本* 是屏幕上可见字符的序列，例如 `Hello, World!`.我们口语上称之为字符串。

Rust实际上有两种表示这段文本的方式： `str` （称为字符串切片）以及 `String`.

A <a href="https://doc.rust-lang.org/std/primitive.str.html" class="external-link" target="_blank">力量</a> 指的是内存中的一个字节序列，它就这么简单。我们与 `&str`，你可能还记得，它是指向内存中字节序列的指针......我们叫 `str` a *字面字符串* 以及一个 `&str` a *字符串切片*.

在与字符串切片交互时，我们只知道极少数几件事：

- 指向字节序列在内存中的位置的指针
- 存储了多少字节
- 任何 `str` 值始终有效，UTF-8
- 每个字节代表一个UTF-8数据点  
  …就这样。我们从 <a href="02-进入原始模式.md" class="internal-link" data-note-icon="" target="">第二章</a> 什么是字节，以及字节序列可能是什么。我们也知道什么是指针，我们把它比作一张藏宝的地图。

让我们在以下Rust Playground中探讨这个问题：

``` rust
fn main() {
    let slice: &str = "Hello, World";  
    dbg!(slice.as_ptr());              
    dbg!(slice.len());                 
    dbg!(slice.as_bytes());            
}
```

什么是UTF-8？我们先退一步。在第二章中，我们从（QWERTY或类似）键盘上的符号转向了一种叫ASCII的东西，基本上是内存中的数字和屏幕上的字符之间的简单映射。然而，即使只是表面上看一眼 <a href="https://www.asciitable.com/" class="external-link" target="_blank">ASCII表</a> 这表明这种映射并不足以覆盖所有类型的情况。举个例子，作为德国人，我缺少了“söme precióus ümlaüts”。

所以我们需要一份更长的文本中可能出现的内容清单。其中一个列表叫做 <a href="https://home.unicode.org/" class="external-link" target="_blank">Unicode</a>这相当全面，也是当今事实上的标准。<sup><a href="#fn1" id="fnref1">[1]</a></sup>

有了 Unicode 列表，我们标准化了转换为二进制（0 和 1）的过程，以便其他人能够普遍地解释这些位。

这个过程，称为 *编码*Unicode 有多种形式，包括 UTF-8。Unicode列表中的一项可以被编码成UTF-8，最多4字节，所以最后那个要点试图告诉我们的是：一个字节 `str` 是一个字节，最多可能属于另外三个字节，用于映射到Unicode列表中的某个条目。

UTF-8 有一个有趣的特性：它完全包含了 ASCII。这意味着 ASCII 表中的任何字符在用 ASCII 编码和用 UTF-8 编码时，字节表示都是相同的。这就是为什么尽管Rust在底层使用了UTF-8，我们仍能在早期章节专注于ASCII技术。

顺便说一句，如果你现在觉得有些事情不对劲 *挺好的* 这和文本的运作方式相符——你可能说得有道理。我们稍后会回到这个话题。

正如你所想，这种数据结构非常高效。如果你想修改它，操作起来也非常困难。本质上，你将被迫重建一个 `str` 每次你都想延长它。

那就是 `String` 进来了。`String` 是真 `struct` 并实现了函数。在引擎盖下，它保持了可生长的特性 `struct` （我们将在本章稍后亲自见到他们）以包含内容，因此修改了 `String` 很简单。本质上，它承担了所有繁重的工作 `str` 我之前暗示过的，是为我们而设的。

最棒的部分是？ `String` 实现特定特征（`Deref<target=str>`准确来说，这个特性确保你能将 `String` 变成了 `&str` *只要不去引用它* （我们通常用&符号来做这件事）。这意味着我们不必担心是否接受 `&str` 或者说，是 `&String` 或者如何将其中一种转换成另一种。  
这很棒，因为这意味着我们可以继续合作 `&str` 总是这样，并且只特别要求函数传递 `String` 必要时。

让我们看看实际操作。

``` rust
fn prints_str(str: &str) {
    println!("I only print &strs, and the &str I got is: {str}");
}


fn main() {
    let slice: &str = "Hello!";
    prints_str(slice);
    
    let mut string: String = String::from("Hello!");
    string.pop();
    string.push_str(", World!");
    prints_str(&string);
}
```

## 引入缓冲区

以下是我们在本章中将逐步铺垫的内容：

- `Editor`：该组件主要在不同的用户界面组件之间进行调解，传递注意力和信息。目前，我们只有全尺寸的 `View`.
- `View`：该组件接收来自 `Editor`，比如字符按键、换行等。该 `View` 利用这些信息提升渲染效率，并将修改事件传递给 `Buffer`.
- `Buffer`：这个组件将保存我们的文本。我们将从简单的纯文本表示开始，逐步扩展到本教程中以容纳更多数据。

`Buffer` 是文本编辑器中的常见结构，包含修改和显示文本文件所需的一切内容。`View` 与 `Buffer` 交互，将内容渲染到屏幕上。在许多文本编辑器中，你可以轻松地在不同 `Buffer` 之间切换，从而同时打开多个文件。

实现细节 `Buffer` 根据文本编辑器变化。例如， <a href="https://www.nano-editor.org/" class="external-link" target="_blank">Nano</a> 使用简单的方法 `Buffer` 结构中没有文档的其他内部表示。保存文件后，文件内容 `Buffer` （不含用于语法高亮的颜色等数据）被保存到磁盘中。另一方面， <a href="https://www.vim.org/" class="external-link" target="_blank">Vim</a> 具有复杂的内部结构，能够高效处理操作，即使是处理大型文件，并包含一个独立的 `Screen Buffer` 这代表屏幕上当前可见的内容。这些设计选择反映了每位编辑的愿景： `Nano` 设计为一个小巧、轻量级的编辑器（因此得名！），适合快速更新配置文件。 `Vim`而，则被设计为一个功能丰富的完整文本编辑器，非常适合处理冗长的文件。

在我们继续实施 `Buffer`，我们需要满足另一个有用的Rust数据结构。

## “维克多，那个向量是什么？”

我们的缓冲区需要动态调整大小——添加项时增大，移除项时缩小——因此我们需要一种特殊的数据结构。这正是Rust中字符串内部管理的功能，使用一种称为 <a href="https://doc.rust-lang.org/std/vec/struct.Vec.html" class="external-link" target="_blank">向量</a>，或者 `Vec`.在一个 `String`，有一个 `Vec<u8>`——本质上是一个字节向量。正如我们在第三章看到的， `u8` 代表无符号的8位数字，本质上是字节。所以，当你操作字符串时，实际上是在处理一个字节向量。

`Vec` 不仅存储字节，还能让其中的数据以 `&str` 的形式返回；`&str` 是对 UTF-8 编码 Unicode 字符序列的引用。`Vec` 的设计目标之一是高效管理内存。例如，如果初始化一个 `Vec` 后在循环中不断添加元素，最简单的实现可能会在每次添加时重新分配内存，并把已有元素全部搬到新空间，但这种做法效率很低。

我们需要一种能随元素增删而扩容或缩容的数据结构。`String` 在底层使用这种结构，从而允许修改所保存的文本。这个结构叫作<a href="https://doc.rust-lang.org/std/vec/struct.Vec.html" class="external-link" target="_blank">向量</a>，即 `Vec`。`String` 在内部维护一个 `Vec<u8>`：它使用灵活的 `Vec` 保存无符号 8 位整数 `u8`，也就是字节。因此它本质上是一个字节向量，而 `String` 会确保这些字节构成有效的 UTF-8 编码；需要时，你可以得到同样指向这些内存字节的 `&str`。<sup><a href="#fn2" id="fnref2">[2]</a></sup>

`Vec`在内部努力确保内存效率。考虑这样一个场景，你创建了一个新的 `Vec` 并且不断循环添加。在简单的实现中， `Vec` 会为所有旧项目和新项目分配一些内存，并将旧项目复制过去，但在内部， `Vec`确保数据复制不需要频繁发生。

## 理解编程中的内存管理

如果你是从抽象化这些细节的语言过渡过来，记忆管理可能会让人感到压力山大。我们简化一下。

计算机最小的内存单元是比特，表示为0或1。例如，正如我们之前见过的，ASCII字符 `a` 是 `01100001`.将二表示为 （`aa`），它们按顺序排列： `0110000101100001`.了解每个字符的大小和起始点至关重要，因为不同字符的部件可能相邻。

这时，提示就派上用场了。它们就像一张地图，显示数据的起点和大小，使数据检索更加高效。考虑 `aa` 举例：指针只需起始位置和一个字符大小即可定位每个字符 `a`.

添加更多角色，比如 `aaa`，并不简单。如果最后一个背后的记忆 `a` 正在使用中，你需要更多空间。这需要向操作系统请求额外内存以容纳新字符，然后复制之前的字符 `a`然后添加下一个——比单纯使用可用内存要慢。  
Rust 解决了编译器层面常见的内存管理挑战：

- 它防止写入超出分配内存的存储。
- 它会仔细管理指针，避免通过一个指针修改数据、用另一个指针访问过时数据等错误。
- Rust 的所有权模型，包括 Move Semantics，能够有效跟踪和管理指针，减少因过时或冲突数据访问而导致的错误。

⠀相比之下，像C这样的语言需要手动内存管理，而Objective-C则使用引用计数（`retain` 以及 `release`）来管理记忆。Java 和 JavaScript 使用垃圾回收器，自动清理未引用对象。

例如，JavaScript 使用字符串指针数组，掩盖了开发者的复杂性。虽然它看起来像字符串数组，但技术上它其实是一个指向字符串的指针数组，类似于 Rust 的 `Vec<String> `但却没有暴露出潜在的指示。

继续我们的讨论 `Vec`我们来考虑一下Rust的 `Vec` 处理数据：  
向 `Vec` 如果每次都把现有元素复制到新的内存位置，可能会显得效率低下。相反， `Vec` 通过最初分配比当前需要更多的内存来优化这一过程。这个额外的空间允许 `Vec` 这样可以存储额外的元素，而无需每次复制整个数组。只有当这种能力被充分利用时，它才会分配更多空间并复制元素。因此，一个 中的项目数量 `Vec` 并不能直接表示它使用的内存量;分配的内存通常超过提升性能的即时需求。该策略平衡内存使用和处理效率。

让我们来讨论两种不同的获取方法 `Vec` 现在。上面代码示例中的变体1只有在你传递的索引是有效的情况下才有效。否则，程序执行将停止：程序会陷入恐慌。

我们需要用代码表示一个函数 *可能* 返回某物——此处指的是 `Vec` - 或者不行。我们已经遇到了一个听起来很适合使用的概念：一个 `Result`，返回一个框，内容如下 `Ok` 如果那里有东西（放这个） *什么东西* 进入盒子），或者一个盒子写着 `Err` 如果发生错误，则将错误放入该信息。

然而，这里的语义不同：它说“这里应该有东西，而没有存在是错误”，而我们想表达的是“可能什么都没有是正常的”。这就是一个 <a href="https://doc.rust-lang.org/std/option/" class="external-link" target="_blank">选项</a> 代表。它的工作原理几乎和 `Result`.坚持用盒子的比喻：从一个 `Vec` 你会退回一个标有标签的盒子 `Some`，上面写着条目，或者一个盒子（其实是一张纸条）写着 `None`意味着 `Vec` 就在那个位置。

如果你想深入了解——这里有一个例子，这次直接摘自文档：

``` rust
#![allow(unused)]
fn main() {
    fn divide(numerator: f64, denominator: f64) -> Option<f64> {
        if denominator == 0.0 {
            None
        } else {
            Some(numerator / denominator)
        }
    }
    
    // The return value of the function is an option
    let result = divide(2.0, 3.0);
    
    // Pattern match to retrieve the value
    match result {
        // The division was valid
        Some(x) => println!("Result: {x}"),
        // The division was invalid
        None    => println!("Cannot divide by 0"),
    }
}
```

## 理解Rust程序的执行与恐慌

当你运行Rust程序时，一切都从以下开始 `main` 函数，其中每次后续调用都会为 *调用栈*.这个堆栈本质上是记录哪些函数处于激活状态及其当前状态——你可以把它想象成一叠实际的纸张，每张纸都描述每个函数的内容。这里有一个简单的设置：

``` rust
fn function_a() {
    function_b();
}

fn function_b() {
    function_c();
}

fn function_c() {
    println!("Inside function_c");
}

fn main() {
    function_a();
}
```

在这个例子中， `main` 叫声 `function_a`，随后称为 `function_b`，导致 `function_c` 这会输出一条信息。每个函数调用都会生成一个新的 *框架* 在调用栈上，且当每个函数完成时，其 *框架* 被移除，回溯到 `main`.换句话说：当我们继续前进 `main()` 到 `function_c()`，堆栈首先通过一张描述上下文的纸条来扩展 `main()`，则 `function_a()`，则 `function_b()`，则 `function_c()`，则由生成的函数 `println!`文本打印出来后，一张张纸被逐张从堆中取出，直到我们重新开始 `main()` 然后结束节目。

### 处理错误 `panic!`

Rust用途 `panic!` 对于它不知道如何处理的错误。默认动作是开始 *展开那叠，* 这意味着要在处理链条上（或翻阅一堆文件）时进行清理。清理工作的一部分是打印关于调用栈的信息（在我们看来，这会显示为 *栈跟踪*）以及变量和结构体的处理。  
或者，你可以配置 Rust 在发生紧急启动时立即中止执行，这样可以让你的程序变小，因为它跳过了回转过程。你在你的 `Cargo.toml` 其中 `panic = 'abort'`.这种方法更快，但要小心——可能会让一些资源如未开文件或网络连接挂机。

### 定制Panic上发生的事情

Rust 为你提供了应对恐慌的多种选择：

- **捕捉放松：** 你在放松的过程中可能会感到慌乱，从而让程序继续运行。这部分是用 `std::panic::catch_unwind`.

- **创建自定义恐慌处理器：** 如果你设计的软件不能直接把控制权交还给操作系统（比如嵌入式系统），你可以明确定义紧急情况下会发生什么。

  以下是定制紧急处理机可能的样子：

  ``` rust
  #[panic_handler]
  fn panic(info: &PanicInfo) -> ! {
      loop {}
  }
  ```

  这段代码说一旦发生恐慌，程序就会无限循环——基本上是冻结，表示为 `->` （意思是：该函数绝不能返回）

- **恐慌钩：** 在恐慌爆发前，你可以设置功能启动来清理，从而为更受控的撞车或清理做准备。

## 何时该慌张

正如我们之前讨论的，通常更倾向于本地处理错误或干净地传播错误，而不是导致程序出现 `panic!`.然而，在防御性编程的语境下，这一规则有例外。

防御性编程意味着预判并防范突发状况。比如我们现在的 `Buffer` 实现假设其 `String`s 从不包含换行。由于我们具体的实现，这一假设目前成立，但未来的修改（也许是我们自己修改）可能会引入导致行为不明确的错误。

为了防止这种情况，我们应该在调试构建中实现“恐慌”的检查——但不允许在发布版本中出现。建议这样做以避免将函数内部文件泄露到函数签名中，并在最终版本中进行不必要的检查。

## 结构生命周期

让我们稍微聚焦一下Rust的生命周期机制。毫不意外，生命周期的阶段是：创造、使用和毁灭。

### 成立

在 Rust 里创建变量和结构相当简单。对于结构体，我们已经大量使用 default（） 了。默认旨在提供一个简单的零参数构造器，返回一个默认类型且无需任何配置。

更复杂的设置逻辑，比如启动网络连接或将终端置于RAW模式，不应进入默认状态。惯用的做法通常是提供一个叫 new 的方法，如果需要可以预期参数，然后执行我们需要的设置，以便完全使用我们的结构。

### 用途

本节将涵盖一个重要的Rust概念，在本教程中我们几乎不会用到。如果你只是为了获取与建造赫克托相关的部分，跳过是安全的。

到目前为止，我们的代码只使用简单的使用模式：我们创建所需内容，函数完成后，Rust 会处理掉所有剩余变量。我们之所以这么容易，其中一个原因是我们还没有大量使用推荐信——我们大多只是根据需要复制，这对我们来说完全可以接受。涉及推荐信时情况会更复杂。正如我们之前了解到的，Rust 确保了 *内存安全*这意味着我们使用的引用总是有效的。它还确保不会发生内存泄漏，这意味着分配给我们的所有内存数据都是真正需要的，且没有分配到任何数据永远不会被程序的任何部分使用。

Rust 确保这一点的部分原因是，在编译过程中确保我们处理的每个变量的寿命。让我们来看一个这种做法行不通的例子：

``` rust
fn longest(x: &str, y: &str) -> &str {
    if x.len() > y.len() {
        x
    } else {
        y
    }
}
```

该代码不编译。之所以不行，是因为Rust无法在编译时判断 a 有多长 `&str` 返回 `longest` 将活着，因此 `longest()` 无法可靠地输出 `&str` 寿命可预测。解决这个问题的一种方法是明确定义 的寿命 `x` 以及 `y`.以下是我们如何执行该规则的方法 `x` 以及 `y` 具有相同的寿命：

``` rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() {
        x
    } else {
        y
    }
}
```

该 `’a` 这里是一个通用的终身规定符，这个上下文可以理解为只接受两个 `&str` s，其寿命相同，返回另一个 `&str` 而且是同一辈子。因此，我们把确保正确寿命的责任放在来电者身上。

还有终身规格 `'static`表示这些数据会在整个程序期间保存，并且有 `'_` 这是匿名的生命周期——不需要命名，因为编译器可以推断出来。

### 毁灭

Rust在生命周期结束后会从记忆中移除内容。不过，如果 `struct` 我们的 OUR MOD 会丢失内存——比如再次禁用 RAW 模式。这就是 `Drop` 特征体现在：如果实现了，其 `drop()` 当结构体退出内存时，方法将被调用。

这带来了一个重要的补充或陷阱： `Drop` 是 *还有* 在慌乱期间调用，这意味着即使程序崩溃，清理活动也会被执行。然而，代码 `drop()` 可以 *还有* 恐慌，如果发生，你将面临“双重恐慌”，迫使拉斯特停止一切并中止。因此，确保你的清理代码万无一失，以避免此类情况。

## 替代屏幕

当我们推理生命周期时 `hecto`我们也来回顾一下我们是如何布置航站楼的。我们可以进一步完善 `hecto` 通过终端的“备用屏幕”功能，这是一项古老的功能，可以保存当前的屏幕缓冲区并切换到我们可用的新屏。退出另一个屏幕后，原始屏幕会恢复。这可以通过以下方式来管理 `EnterAlternateScreen` 以及 `LeaveAlternateScreen` 来自 <a href="https://docs.rs/crossterm/latest/crossterm/terminal/struct.EnterAlternateScreen.html" class="external-link" target="_blank">跨项</a>.作为开发者，我们最大的一个好处是 `clippy` 警告内容会被保留，不会从屏幕上消失;它们在退出时依然可用 `hecto`.

这与我们之前讨论过的Lifetimes有关。在这种情况下，我们需要确保有人 *否则* 获得的推荐信有效时间取决于 *他们* 即使在我们结构生命结束后，也需要它。

让我们来分析一下为什么在设置恐慌钩时使用盒子至关重要。如果不这样做，我们会定义闭合并传递给 `set_hook`.然而，在某个时刻，我们的代码可能会丢失对该闭包的所有引用，导致该闭包被重新分配。这很有问题，因为紧急钩子必须持续整个程序生命周期——甚至可能远超我们具体的期限 `Editor` 目标。

为了确保关闭的存续，我们需要将其所有权转移到 `set_hook`.我们通过将其放入 <a href="https://doc.rust-lang.org/std/boxed/struct.Box.html" class="external-link" target="_blank">盒子</a>.这会在内存中“框框”闭包，我们传递指向该盒闭包的指针到 `set_hook`.现在，紧急钩子函数拥有盒子内的数据，确保即使原始代码消失，闭包依然存在。

顺便说一句，这也强化了我们设计决策的必要性 `Editor` 从程序开始到结束的唯一数据结构<sup><a href="#fn1" id="fnref1:1">[1:1]</a></sup>

现在给出上面同一代码的注释版本：

``` rust
//Retrieve the current hook, which by default does some nice printing of the panic
let current_hook = std::panic::take_hook(); 
// Define a new closure that takes a reference to the PanicInfo.
// Move any external variables needed within the closure here. 
// Place the closure into a Box and set it as the new panic hook.
std::panic::set_hook(Box::new(move |panic_info| {
    // Our custom panic hook logic goes here
    // Execute the original hook to retain default panic output behavior.
    current_hook(panic_info);
}));
```

第一种是，通过实现例如 from（），你就能免费获得相应的 into（） 功能。换句话说：如果你实现 `from` 对于类型 `A` 以及 `B`你们俩都可以打电话 `B::from(a)` 以及 `a.into()` 转身 `a` 变成 `B`.  
第二个是Rust可以用这个特性为我们做隐性转换。这最好用代码来理解，所以这里有一个Rust Playground示例来说明了这个概念：

``` rust
#[derive(Debug)]
struct ErrorA;
#[derive(Debug)]
struct ErrorB;

impl From<ErrorB> for ErrorA { //We allow converting ErrorB into ErrorA here
    fn from(_: ErrorB) -> ErrorA {
        ErrorA{} // We would typically carry over some information from ErrorB into ErrorA. We don't do this here because we want to illustrate something else.
    }
}

fn do_something_else() -> Result<(), ErrorB> {
    Ok(())
}

fn do_something() -> Result<(), ErrorA> {
    do_something_else()?; // here is where the implicit conversion happens: Even though this function returns an ErrorB upon failure, Rust will use the from implementation to convert it into ErrorA and propagate it up.
    Ok(())
}

fn main() {
    do_something().unwrap();
}
```

让我们先描述一下 `core`. `core`是Rust的库，包含绝对的本质，如 `Option`, `Result`、宏量营养素和其他东西。它适用于任何Rust环境。 `core`它不假设你需要操作系统支持，因为你可能是自己构建操作系统！

`std` 包含所有内容 `core`但还会添加大量其他假设操作系统级功能的内容，比如线程、网络、堆内存分配、文件I/O等。基本上， `std` 是 `core` 此外，还有这些标准的操作系统相关功能。为了消除去弄清楚真相的负担 `core`以及什么是 `std`, `std` 重新导出所有内容 `core`.

对于非标准环境，例如嵌入式系统，编译可以通过设置实现 `#![no_std]` main.rs，但这也要求你的依赖关系在没有 S 的情况下工作 `std`.所以如果你将来打算从事嵌入式系统工作，现在你已经知道如何在那里制作表情符号了！

## 石墨簇宽度

说到表情符号：确实如此 *不是* 还没完全发挥作用。任何包含表情符号的行仍然会有滚动问题，尽管我们尽了最大努力：你无法完全滚动到那行的末尾。  
这同样适用于某些其他字素簇。我们现在已经纠正了 `hecto` 所以一条线的“长度”指的是屏幕上可见的字素簇数量，但我们还没考虑到的是，字素簇的宽度可以不同！

Unicode 中的宽度有以下几种类型：

- **零宽度字符：** 像零宽度连接器（ZWJ，我们之前见过这个）这样的字符用于文本格式化，且没有宽度，意味着它们在视觉上不占用空间。同样的道理（某种程度上）适用于不可打印的角色。由于它们不可打印，所以没有宽度。
- **半宽（窄）字符：** 代表标准可打印字符，在单宽环境中如终端中，这些字符宽度通常为1。这是基础文本处理中最常见的情形。
- **全宽（宽）字符** 这包括来自复杂文字的字符，如中文、日文和韩文。它们是双宽的，因为它们占据了单宽字符网格中的两个空格，这对于文本渲染中的正确对齐至关重要。以下是拉丁字母的全宽字符：ABCDEFGHIJKLMNOPQRSTUVWXYZ。
- **模糊：** 该类别的字符通常根据所处的语境或字体，分为窄或宽。例如，某些符号，比如中点 `·` 或各种划号 `-`根据字体的不同，可以被表示为窄或宽。
- **中立：** 这些角色通常占据 *有些* 但具体多少取决于上下文或字体。表情符号属于这一类，空格和标签也属于。

我们这里需要另一个笼子来帮忙： `unicode-width`.让我们来Rust Playground见面它：

``` rust
use unicode_width::UnicodeWidthStr;
fn main() {
    let zero = "\x1b";              // Esc, see Chapter 3
    let half = "Ä";                 // German A with Umlaut
    let full = "Ａ";                // Like an A, but wider
    let neutral_1 = "👋";           // Normal Emoji
    let neutral_2 = "🧑🏿‍🤝‍🧑🏿"; // Compound Emoji
    let ambiguous = "·";            // Mid Dot
    println!("Type\tChar\tWidth\tWidth (CJK)");
    println!("Zero\t{zero}\t{}\t{}", zero.width(), zero.width_cjk());
    println!("Half\t{half}\t{}\t{}", half.width(), half.width_cjk());
    println!("Full\t{full}\t{}\t{}", full.width(), full.width_cjk());
    println!("N(1)\t{neutral_1}\t{}\t{}", neutral_1.width(), neutral_1.width_cjk());
    println!("N(2)\t{neutral_2}\t{}\t{}", neutral_2.width(), neutral_2.width_cjk()); 
    println!("A \t{ambiguous}\t{}\t{}", ambiguous.width(), ambiguous.width_cjk()); 
}
```

`cjk` 代表中文、日文和韩文，因为这些文字很可能包含模糊宽度的字符。

表情符号依然让我们头疼。好消息是，“正常”表情符号通常可以被视为全宽。坏消息是复合表情符号的使用权已经完全失效。原因是表情符号的渲染方式取决于字体：如果支持复合表情符号，则以宽字符形式呈现;如果不支持，则渲染所有部分。Rust Playground 很好地展示了这一点，因为它使用了不同的字体——输入字段不支持复合表情符号，输出字段支持。

## 字符宽度 `hecto`

hecto只支持窄字符和宽字符。

**零宽度字符** 将以狭窄替换字符渲染。这让与他们互动和导航变得更方便。

我们的策略 **模棱两可的人物** 将遵循Unicode的建议。建议在非日韩语语境中，或无法可靠确定语境时，将这些模糊字符视为狭义，所以我将采用 `width`代替 `width_cjk` 往前走......

对于 **中立角色**, `unicode-width` 我会做出有根据的猜测，不管是宽义、狭义还是完全不同。如果是0，我们就按上述方式处理。如果超过2，我们就把它当作宽字符。这意味着 `hecto` 在  
不支持复合字符的字体。

# 任务13：支持字符宽度

在这次作业中，我们将正确支持字符宽度。根据你的代码对我实现步骤的遵循程度，你的具体步骤会有所不同，但最终目标是一致的：我们希望正确支持不同宽度的字符。

我们的策略是替换 `String` 存储完整的字符串 `Line` 对于每个字素自定义数据结构，存储字素本身，其宽度近似为 `hecto` 以及潜在的替代者。我们会尝试为任何无法在屏幕上正确显示的角色挑选一个合适的替代角色，无论是因为宽度为零，还是像贝尔这样的控制型角色。

这个结构看起来像这样：

``` rust
enum GraphemeWidth {
    Half,
    Full
}

struct TextFragment {
    grapheme: String,
    rendered_width: GraphemeWidth,
    replacement: Option<char>
}
```

我们需要更新 `from` 对输入字符串的字素簇进行迭代，生成我们的内部 `Vec<TextFragment>`.对于角色来说 `unicode-width` 识别宽度为0，我们将添加一个替换字符来显示，而不是实际的字素。在这个作业中，我们只用中点替换任何0宽度的字素（`·`).我们稍后会加入更复杂的操控。

我们的 `get` 函数需要返回在指定范围内可见的字形。为此，我们需要遍历向量，跟踪每个处理字符的宽度，并返回起始和结束于指定范围的字素。我们还需要特别注意屏幕边界：如果宽字符被屏幕边缘截断，我们会打印水平省略号（`⋯`）以表示发生了截断。

我们对 `Location`， a `Position` 偏移量需要稍作调整。 `Location` 现在指向某行中的单个字素。 `scroll_offset`而 则不再指代文本中的某个位置——它指的是无限格子上的一个点，每个字素可以填满其中一到两个格子。或者简单来说：如果我们想象我们的终端无限延伸， `scroll_offset` 指的是该无尽终端上的一行和一列。这意味着a） `Position` 不再仅描述可见屏幕上的位置，而是描述该网格上的位置，b） `scroll_offset`将不再是 `Location`，但 `Position`c）我们需要一种方法来处理这些结构，以处理插入点的位置、文档内的移动以及正确的滚动操作。

为此，我们需要一种方法来转换 `Location` 变成了 `Position`.我们通过将电流之前的所有宽度相加来实现 `x` 给定的 `Location`，着陆到正确的格子。由于文档中的每一行恰好占据网格上的一列，我们可以简单地取值 `y` 以确定正确的行。

随着这个变化，我们的滚动逻辑需要转换 `Location` 转给 `Position` 并对此进行判断，确定 Cart 是否移出了屏幕，并进行更新 `scroll_offset` 相应地。为了确定滑针在屏幕上的位置，我们需要减去 `Position` 当前字素在网格上的 `scroll_offset`.

我们不需要动逻辑就能继续阅读文本。这意味着如果线条宽度不同，我们不会总是保持正确的垂直位置。但我们要么期望使用一致的宽度，要么只偶尔使用表情符号，所以在大多数情况下，从一行切换到下一条是可以接受的。

# 作业14：文本查看器

让我们用最后的润色来结束这一章，并改进对隐形角色的处理。目前，我们正在用中点替换部分，但不是全部。我们对一些零宽度字符的处理也有些奇怪：如果你有一串10个零宽度连接器， `unicode-segmentation` 识别它们为 *一* 因此我们用 替换 *一* 中点。  
然而， `hecto` 目标是成为文本编辑器，主要重点是显示和编辑可见的文本。如果有人想编辑隐形文字， `hecto` 不适合他们。

不过，输出隐形字符可能会对终端产生奇怪的副作用。例如，一个Tab（`\t`）可能会占用终端上奇怪的空间，导致视觉故障。如果我们不替换一个转义字符，正如我们了解到的，这个字符在ASCII和Unicode中都是完全有效的，它就会直接被发送到终端，作为潜在的转义代码，触发一些奇怪的终端行为。顺便说一句，这就是为什么如果你错误地将二进制文件以文本形式输出到终端，终端往往会完全出问题。

制表表表现得如此奇怪的原因又让我们回到机械打字机：制表表——物理阻挡器，防止带纸张的滚筒超过某个点——对于保证文本一致性至关重要，比如在打简历时。由于Tabs非常古老，它们的处理方式会根据出现地点不一致。

Unicode为我们提供了丰富的替代字符，如果我们感兴趣的话。ASCII衬衫里所有“无法打印”的东西都在那里，包括我们的老朋友␇和␛。 <a href="https://www.unicode.org/charts/PDF/U2400.pdf" class="external-link" target="_blank">你可以在这里找到完整名单</a>.

不过，我们的目标并不是 *完备性* 在替换角色时，我们采取更简单的方法。

这是作业内容：

- A Space 应该渲染为 Space——这一点保持不变。
- 更换任何标签（`\t`）后带有空格。
- 将每隔一个可见的空白空间（即长度非零的空白空间）替换为 <a href="https://www.compart.com/en/unicode/U+2423" class="external-link" target="_blank">开箱</a> (`␣`).要判断一个字素簇是否是空白，请 `trim()` 在上面去除所有空白，然后再检查一下。 `is_emtpy()` 如果还有什么留下的话。
- 对于像Bell或Null这样的控制字符，使用零宽度字符替换为 <a href="https://www.compart.com/en/unicode/U+25AF" class="external-link" target="_blank">白色竖直矩形</a> (`▯`).你可以使用 `is_control()` 对一个角色进行判断是否是控制角色，你可以将两者都变成 `String` 以及一个 `str` 通过调用，使其成为其特征的迭代器 `.chars()` 马上去。
- 对于其他零宽度角色，我们会不断用中点替换它们。
