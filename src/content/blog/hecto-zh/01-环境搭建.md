# 第 1 章：环境搭建

## 第一章：布置

<img src="/images/hecto-zh/flenker-1712399493-0.png" alt="flenker-1712399493-0.png|philipp&#39;s blog" />

啊，第一步。你难道不喜欢在一张白纸上重新开始吗？然后选定那块你将建造整个宫殿庄园的砖块？

不幸的是，当你在建造一个 **计算机项目**，第一步可以......复杂。而且令人沮丧。你必须确保你的环境是为你使用的编程语言设置的，并且要弄清楚如何在那个环境中编译和运行你的程序。

幸运的是，Rust自带的开发环境已经帮我们完成了大部分工作，所以你只需要一个文本编辑器<sup><a href="#fn1" id="fnref1">[1]</a></sup> 以及 `rust`.这个教程是在Mac上写的，不过 *应该* Windows和Linux也能用。

## 如何安装Rust

为了安装Rust，我们将使用 `rustup`管理已安装的 Rust 版本及相关工具。  
《Rust》附带一本精彩且免费的书： <a href="https://doc.rust-lang.org/book/title-page.html" class="external-link" target="_blank">Rust编程语言</a>.每当我觉得这本书解释得比我更清楚时，我会经常链接到这本书以获取更深入的了解。安装 `rustup`，跟着我走 <a href="https://doc.rust-lang.org/book/ch01-01-installation.html" class="external-link" target="_blank">第1.1章：就职</a> 为了你的平台。撰写本文时，Mac的操作方式如下：打开一个终端（在Spotlight中输入“Terminal”找到它），并输入以下命令：

    curl --proto '=https' --tlsv1.2 https://sh.rustup.rs -sSf | sh

安装程序会给你一些选项，并建议你按默认设置 `Enter`.装置以以下鼓励的话语结束：

    Rust is installed now. Great!

重启终端，然后输入以下内容：

    rustc --version

你应该会看到关于你安装版本的一些信息。如果没有，可以前往 <a href="https://doc.rust-lang.org/book/ch01-01-installation.html" class="external-link" target="_blank">故障排除安装指南</a>.

顺便说一下，这个教程我们会大量使用终端，所以保持它开着就行。

## 该 `main()` 功能

创建一个名为 `hecto.rs`然后给它一个 `main()` 功能。(`hecto` 这是我们正在构建的文本编辑器的名称）。

``` rust
fn main() {
    return;
}
```

在 Rust 里，你必须把所有可执行代码都放进函数里。该 `main()` Rust 的功能很特殊。它是运行程序时默认的起点。当你从那个地方回来时 `main()` 函数，程序退出并将控制权交还给操作系统。<sup><a href="#fn2" id="fnref2">[2]</a></sup>

Rust 是一种编译语言。这意味着我们需要通过Rust编译器运行程序，将其转为可执行文件。然后我们像运行命令行其他程序一样运行该可执行文件。

要编译 `hecto.rs`，请在终端中运行 `rustc hecto.rs`。如果没有报错，它会生成一个名为 `hecto` 的可执行文件。输入 `./hecto` 并按 `Enter` 即可运行；这个程序不会打印任何输出。<sup><a href="#fn3" id="fnref3">[3]</a></sup>

## 使用 `cargo` 编译

理论上，构建 `hecto` 只需要 `rustc`。但在实际开发中，Rust 还提供了一些便利工具，能让开发过程更轻松，而且正如稍后会看到的，也更安全。

Rust 自带一个名为 `cargo` 的程序。如果你熟悉 `JavaScript` 生态系统，可以把 `cargo` 理解为 Rust 版本的 `npm`。它能管理依赖、编译代码，并完成其他项目任务。正确安装 Rust 后，可以输入以下命令调用 `cargo`：

    cargo --version

我们将用这个程序开始创建 `hecto`。输入 `rm hecto.rs hecto`，删除之前的 `hecto.rs` 文件和 `hecto` 可执行文件。

在你想要的目录里 `hecto` 要激活，请输入以下内容：

    cargo init hecto --vcs none

使用 `--vcs none` 部分确保你启动 `hecto` 没有 `git` 支持。如果你想使用 `git` 在这个项目中，省略这个标志（但那样你就需要 `git` 已安装）

`cargo` 通过告诉你以下内容来宣称成功：

     Created binary (application) package

它创建了一个名为 `hecto`，填充了一些文件。我们稍后会详细介绍大部分，其中最有趣的是一个名为 `main.rs` 在一个叫做 `src`.

我们来看看那个。打开它时，你会发现里面已经包含了 `main()`.仅凭观察，我们可以推断这个程序 <a href="https://en.wikipedia.org/wiki/%22Hello,_World!%22_program" class="external-link" target="_blank">遵循古人的仪式</a> 通过打印“Hello， World！”然后离开。

使用 `cargo` 要编译该程序，请确保你处于 `hecto` 文件夹（运行） `cd hecto` 之后 `cargo init hecto`），然后运行 `cargo build` 在你的壳里。 `cargo` 不无声，因为 `rustc`曾经和将来产生的输出看起来有点像：

       Compiling hecto v0.1.0 (/Users/pflenker/repositories/hecto)
        Finished dev [unoptimized + debuginfo] target(s) in 12.36s

执行这个命令会添加一些额外的文件（我们稍后会详细说明），并会生成一个名为 `hecto` 放进文件夹 `target/debug`.执行吧 `./target/debug/hecto` 说服自己这个程序确实会打印出来 `Hello, World!` 然后离开。

## 理解 `cargo`的额外文件（及特色）

我们的 `hecto` 目录现在包含了很多东西（有些是隐藏的，具体内容在你的机器上可能不同）：

1.  一个文件夹，叫 `src`，包含 `main.rs`.
2.  两个文件命名 `Cargo.toml` 以及 `Cargo.lock`
3.  一个文件夹，叫 `target` 里面有一些隐藏文件和一个叫做 `debug`.那个文件夹里有更多文件、文件夹和我们的可执行文件， `hecto`.  
    这些额外的东西有什么用？让我们学习吧。

### 该 `src` 文件夹

该 `src` 文件夹是我们所有源文件存放的地方。现在，我们只有一个。很快，会有更多。

### `Cargo.toml` 以及 `Cargo.lock`

该文件遵循一种称为 <a href="https://toml.io/en/" class="external-link" target="_blank">TOML</a> 并用来告诉编译器它应该编译的代码的一些信息。我们仔细看看几段前的默认版本......

`cargo` 我们也做依赖管理，还有 `Cargo.toml` 会帮我们保管这些依赖关系。该 `Cargo.lock` 是依赖管理的一部分，确保依赖在不同环境中保持一致。现在不用担心这个——我们会在下一章深入探讨。

### 构建目标

让我们来看看 `target` 以及其内容。  
`cargo` 支持多个所谓的 *构建目标*.默认使用的是 `debug`这意味着最终的可执行文件主要面向我们开发者，而非真正的终端客户。

另一个有效的目标是 `release`，这恰恰相反：要把东西交到客户手中，而不仅仅是开发者。

要构建发布版本，请运行：

    cargo build --release

一旦你这样做，你会注意到另一个文件夹出现在 `target` 文件夹，名副其实 `release`.

等等，“可执行文件针对某人”是什么意思，难道不只打印“Hello World”吗，不管是谁用的？<sup><a href="#fn4" id="fnref4">[4]</a></sup>

确实如此。  
首先，Rust编译器试图对开发者和终端用户非常宽容，因此它对 `debug` 以及 `release` 构建方式不同。我们稍后会遇到一个具体的例子，但有几种编程错误，Rust 会假设你真的、真的不是故意这么做的。如果 `debug` 构建时遇到这种错误，程序崩溃，这是Rust能告诉你你错得有多严重的例子。但Rust *确实* 有办法从这些错误中恢复并继续。虽然结果不对，但确实如此 *可以* 继续，假设对终端用户来说，错误的结果是坏事，但崩溃更严重，同样的错误不会让 `release` 建造。

其次，编译器可以在底层进行各种优化，以加快结果速度。但这些优化需要时间，也使编译速度变慢。谁会一直在编译？开发者。谁从不编译？用户。因此， `debug` 构建会关闭优化以优先考虑开发速度， `release` 构建让玩家优先考虑执行速度。

最后，可执行文件确实包含除了“Hello， World”之外的代码，Rust 会为 `debug` build 能帮你更快地调试问题。这些信息对客户来说不需要，因此可以从发布版本中排除。

`cargo` 在合辑中也告诉我们这一点，最后一句话 `debug` 新闻稿内容如下：

    Finished dev [unoptimized + debuginfo] target(s) in 0.04s

这说明构建未经过优化，包含调试信息。

该 `release` 而 build 则以以下内容结束其合集：

    Finished release [optimized] target(s) in 0.04s

这告诉我们没有包含调试信息，而且这是一个优化版本。

让我们仔细看看调试信息，但别动我们新开发的、已经很受欢迎的文本编辑器代码。相反，直接去 <a href="https://play.rust-lang.org/?version=stable&amp;mode=debug&amp;edition=2021" class="external-link" target="_blank">Rust Playground</a>，这个函数已经包含了我们的Hello World函数。在左上角，你会看到一些按钮：调试，然后是稳定，然后 `...`.点击这三个点并启用 `backtrace`.

然后在游乐场中更改代码如下：

``` rust
fn main() {
   panic!("Hello, World");
}
```

`panic!` 引发崩溃，简单明了。点击“运行”即可运行。然后点击“调试”，改成“发布”，再运行一次。

你会注意到`release` 构建生成以下输出：

         Running `target/release/playground`
    thread 'main' panicked at src/main.rs:2:4:
    Hello, World
    stack backtrace:
       0: rust_begin_unwind
       1: core::panicking::panic_fmt
       2: playground::main

该 `debug` 输出如下：

         Running `target/debug/playground`
    thread 'main' panicked at src/main.rs:2:4:
    Hello, World
    stack backtrace:
       0: rust_begin_unwind
                 at /rustc/7cf61ebde7b22796c69757901dd346d0fe70bd97/library/std/src/panicking.rs:647:5
       1: core::panicking::panic_fmt
                 at /rustc/7cf61ebde7b22796c69757901dd346d0fe70bd97/library/core/src/panicking.rs:72:14
       2: playground::main
                 at ./src/main.rs:2:4
       3: core::ops::function::FnOnce::call_once
                 at /rustc/7cf61ebde7b22796c69757901dd346d0fe70bd97/library/core/src/ops/function.rs:250:5

我们还没有详细讨论堆栈追踪，但即使没有这些知识，你也能立刻看到调试输出中包含的信息比发布版本更多。理解 *调用栈*，因此函数调用其他函数及更多函数直到其中一个函数崩溃的顺序非常有用。

### 建造神器

这就解释了文件夹，但里面那些不在里面的东西呢？ `hecto`?让我们调查一些事情以进一步理解。

回到终端，按顺序执行以下命令：

    cargo clean
    cargo build
    cargo build

（是的，我们跑 `cargo build` 两人）

第一个命令清除 `target` 目录。第二个命令和之前一样运行。不过在第三个阶段， `cargo` 不输出以 开头的行 `Compiling`，就像以前一样。这是因为 `cargo` 可以判断当前版本的 `main.rs` 已经被编译完成。如果 `main.rs` 自上次编译以来未被修改， `cargo` 也没再跑一次合集。如果 `main.rs` 变了，然后 `cargo` 重新编译 `main.rs`.一旦代码库扩展，这会更有用，因为大多数组件在只修改一个组件的源代码时，不需要反复重新编译。

所有（好吧，大多数）额外的文件都在 `target` 目录是 *建造神器* 使得 `cargo` 以便更快地完成后续的合辑。 <a href="https://doc.rust-lang.org/cargo/guide/build-cache.html" class="external-link" target="_blank">更多信息请参见 <code>cargo￼’s</code> 文档。</a>

## 编译与运行

既然你很常见的是编译和运行你的程序， `cargo` 将这两步与命令结合起来 `cargo run`.

试着在 中更改返回值 `main.rs` 到其他字符串 `Hello, World`.那就跑吧 `cargo run`你应该能看到它被编译出来。检查结果，看看你换成的弦是否是那个。然后再改回去 `Hello, World`重新编译，并确保它恢复到返回状态 `Hello, World`.

## 总结与展望

在本章中，我们安装了Rust，启动了一个简陋的项目，并熟悉了它。我们现在知道如何手动编译代码，并且 `cargo` 并且要有良好的初步理解 `cargo` 以及它生成的文件。最重要的是，我们可以建造东西，必要时清理身后。我们还遇到了Rust Playground，这在我们想单独探索新事物后再付诸行动时非常有用。

在 <a href="02-进入原始模式.md" class="internal-link" data-note-icon="" target="">第二章</a>我们要构建一个程序，它读取用户输入，打印到屏幕上，按下按钮后退出 `q`.仅这一点就为我们带来了大量初学者的学习。
