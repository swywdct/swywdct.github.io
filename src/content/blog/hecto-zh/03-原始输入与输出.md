## 了解习用语

在使用像Rust这样的新语言时，你可能听说过 **习语** 代码。这不仅仅是让你的代码正常工作;关键是让它与该语言中通常的做法相契合。但这为什么重要？

我们这样想：假设我告诉你，按照这个教程操作，你可以“用一把苍蝇拍杀死两只苍蝇”——一边学习Rust，一边构建自己的编辑器。你明白我的意思吗？

对一些人来说，比如德语使用者，这可能会让你有印象，因为它和一句常见的德语谚语很相似。也有人会用不同的方式表达，比如葡萄牙语“一击杀死两只兔子”。但对于不熟悉这些说法的人来说，可能需要一些解释。在英语中，我们通常说“一石二鸟”，强调使用人人都能理解的表达的重要性。虽然非成语短语仍能传达重点，但往往导致对措辞的过度关注而忽视信息本身。

写成语代码有点像这样。关键是遵循语言中的常规规则和实践，让你的代码更容易阅读和理解。通常，只有当代码出现问题时，人们才会深入你的代码——要么是有bug，要么是想添加新内容。因此，让你的代码易于理解不仅有助于你当前的合作者，也有助于你未来在调试或添加更多功能时的自己。

我们将分三步来解决：优化代码以适应常规样式，使用 `crossterm` 以正确的方式使用，并从90年代那个熟悉的回形针中获得一些指引，保持正轨。<sup><a href="#fn1" id="fnref1">[1]</a></sup>

在不改变代码工作方式的情况下精炼代码称为 *重构*.我们的目标并不是一开始就完美;而是不断改进，使我们的代码更易于操作和构建。为了确保我们的更改不会破坏任何东西，我们暂时会手动测试。虽然自动化测试很棒，但它们超出了我们本教程中讲述的内容。

### 将代码拆分为多个文件

在 Rust 中，和许多编程语言一样，主方法通常只是作为应用的入口开始，之后不会做太多其他操作。我们的目标是将代码放置在逻辑上合理的位置，以便未来更容易查找和维护。这种方法还有许多其他好处，我会在遇到时指出。

目前，我们的代码有点难以理解。要真正理解它的作用，你必须把整个内容都挖掘出来。简而言之，它会回响用户按下的每一个按键，按下“q”键时才会退出。让我们用一把武器杀死两只动物来简化生活：我们将为编辑引入一个新的结构，名为 `Editor`，该文件将存在于独立文件中。这样代码更容易理解，也更符合习语。

`struct` 基本上是一组变量——有时也是函数——这些变量汇聚在一起形成一个有意义的整体。这正是我们保持整洁有序所需的。

这次代码变更实际上扩展了我们与 `match`，所以是时候理解了 `match` 稍微好一点，还有一些其他概念，顺便说一句。

我们先从内心说起 `match`.它看起来像这样：

``` rust
match (event.code) {
    Char(c) => {
        if c == 'q' {
            break;
        }
    },
    _ => (),   
}   
```

外部 `match` 陈述（我们稍后会讨论）确保此时我们知道 `event`是 `KeyEvent`，有文档 <a href="https://docs.rs/crossterm/latest/crossterm/event/struct.KeyEvent.html" class="external-link" target="_blank">给你</a>:

``` rust
pub struct KeyEvent {
    pub code: KeyCode,
    pub modifiers: KeyModifiers,
    pub kind: KeyEventKind,
    pub state: KeyEventState,
}
```

那么，我们是如何从这种结构中提取出一个角色的呢？回想我们关于错误处理的讨论，我描述了两个隐喻性的盒子：一个粉色的盒子里装着我们想要的内容，一个黑色的盒子里装着错误。我们在这里延伸这个比喻。

让我们来看看 `KeyCode`:

``` rust
pub enum KeyCode {
    Backspace,
    Enter,
   // More KeyCodes
    Char(char),
   // Some more KeyCodes
}
```

<a href="https://docs.rs/crossterm/latest/crossterm/event/enum.KeyCode.html" class="external-link" target="_blank">以下是完整列表</a>，如果你感兴趣的话。所以 `KeyCode` 可以表示：

- a `Backspace`
- 一个 `Enter`
- 还有其他一些东西（可能是钥匙）
- 一个包含字符的盒子
- 额外内容（可能还有更多按键）

那是 *非常大* 有用，去 *方式* 超越了通常的if/then/else构造。什么是 `match` 可以描述为：“如果 `event.code` 看起来像一个装着角色的盒子，我们就叫这个角色吧 `c` 并执行以下方块“。

还有一点需要注意 `match`：匹配需要是穷尽的，意味着必须处理所有可能的情况。你必须有意识地决定该怎么做，以防没有角色的盒子，而是别的东西。我们指导 `match` 处理方式如下： `_ => ()`.这意味着：对于所有不匹配的（记为 `_`），不做任何事（记为 `()`).

你可能会问，这难道不矛盾吗？如果 `match` 设计上明确涵盖所有案例，那为什么要简单地匹配“所有”？说得很好，你其实可以教书 `clippy`我们即将见到他，以强调这是一个潜在的问题。

现在我们更容易理解外部 `match`，其形状如下：

``` rust
match read() {
    Ok(Key(event)) => {
        // ... inner match
        }   
    },
    Err(err) => println!("Error: {}", err),
    _ => ()
}
```

第一分支两次使用了盒子概念。它写道：如果 `read()`返回一个标有粉色标签的盒子 `Ok`，而在那个粉色框里还有另一个标有标签的框 `Key`然后打电话给那个盒子里的东西 `event` 继续。  
第二个分支直接处理错误：如果存在黑箱，取 `err` 拿出来展示。  
最后一个分支处理所有其他组合：对所有粉色 `Ok` 不含标注框的盒子 `Key`什么都别做。

## “看起来你在写程序。需要帮忙吗？”

现在的规范是习语了吗？看起来是这样。但它真的特别吗？不完全是。懂所有成语并不代表你就是诗人。在某种程度上，正如前面提到的，这是可以接受的。不过，我们可以寻求 <a href="https://en.wikipedia.org/wiki/Office_Assistant" class="external-link" target="_blank">30年老的烦人回形针</a> 进一步完善我们的代码： `clippy`.

`clippy` 既是Microsoft Office 97的一个令人恼火的功能名称，也是用来指出我们代码可能改进的工具——一个linter。你可以用以下命令行运行：

    cargo clippy

`clippy` 已经突出了一些刻意的<sup><a href="#fn2" id="fnref2">[2]</a></sup> 问题。这里有一个例子：

    warning: unnecessary parentheses around `match` scrutinee expression
      --> src/editor.rs:19:27
       |
    19 |                     match (event.code) {
       |                           ^          ^
       |
       = note: `#[warn(unused_parens)]` on by default
    help: remove these parentheses
       |
    19 -                     match (event.code) {
    19 +                     match event.code {
       |

花点时间欣赏这些反馈的精准和实用！它特别指出了代码中有问题的部分。我们很快会讨论这个问题，但先让大家先说 `clippy` 进行更全面的检查。 `clippy` 有各种设置或旗帜，而对于我们这些初学者来说，最合适的是 `pedantic`，使得 `clippy` 提出所有可能的成语改进建议。

我们是这样操作的：

    cargo clippy -- -W clippy::all  -W clippy::pedantic

该命令现在会触发多个警告。  
不仅如此 `clippy` 指出我们代码中的弱点，同时还提供了文档链接供进一步阅读。太好了！

`clippy` 也可以帮我们解决一些问题：

    cargo clippy --fix -- -W clippy::all  -W clippy::pedantic

我们能看出来 `clippy` 我们希望默认使用哪些标志，例如在我们的顶部添加以下代码 `main.rs`:

``` rust
#![warnall, clippy::pedantic]
```

该指令激活了这两个类别的警告 `all` 以及 `pedantic`.尽管名字如此， `all` 不包括 *一切*，但有范畴，如 `correctness`, `suspicious`, `style`, `complexity` 以及 `perf`.正在看 <a href="https://doc.rust-lang.org/nightly/clippy/" class="external-link" target="_blank">纪录片</a>，看起来像 `style` 正是我们需要的，但其他人也不会伤害我们。

<a href="https://github.com/pflenker/hecto-tutorial/commit/4a7df66f1103b318b28b8d5374aa172e0bf3ac8a" class="external-link" target="_blank">让我们在这次提交中应用这些更改，包括 clippy 的自动修复和手动修正。</a>.

通过这个过程， `clippy` 教会了我们两个教训：如何优化格式字符串以及使用 `if let `为了更干净的模式匹配。

`clippy` 仍然理所当然地指出我们的 `run()` 功能不依赖于编辑器的具体实例。我们有意这样设计，同时引入不同类型的功能，我们很快会改变这一点。

最终，我们的目标应该是毫无预警地编译代码，向未来的开发者传达我们努力产出最佳代码的信号。每当教程步骤涉及练习时 `clippy` 我会不看好，我会在正文中澄清这一点。

## 偏执 `clippy`

我们可以启动 `clippy` 提升到11级，并让它极度警惕。文档描述了可以启用的各种额外标志以最大化 `clippy`审视。以下命令包含所有看似有用的选项 `hecto`:

    cargo clippy -- -W clippy::all -W clippy::pedantic  -W clippy::nursery -W clippy::cargo -W clippy::restriction

这种方法很严谨，但也有代价：越多疑越多疑 `clippy` 越有可能标记某些看似糟糕的代码，而这些代码在当时情况下实际上是可以接受的。更糟的是，我们启用的一些规则可能会指出完全有效的代码存在问题，或者与其他规则相矛盾。  
以我们目前的经验水平，完全理解这些细微差别可能很有挑战性，我们可能会陷入错误的领域。对于我们这些Rust新手来说，这个吹毛求疵的设定已经很有野心了。所以，我们先说到这里。

## 改进的错误处理

让我们以关于错误处理的承诺来结束对惯用代码的探讨。我们的目标是：

- **本地处理：** 如果我们能及时解决错误，我们会处理。
- **向上传球：** 对于无法直接修复的错误，我们会将其传递到上级链进行进一步处理。
- **顶级操控：** 如果错误达到最高级别，我们会尽量优雅地处理。

## 清空屏幕

我们即将深入探讨每次按下按键时，编辑器的用户界面如何渲染。这发生在三个关键时刻：

- **启动：** 我们正在为用户铺设舞台。
- **每次按键后：** 以回应用户的行为。
- **退出前：** 整理我们的工作区，留下整洁的告别信息。

首先，我们先清理屏幕。我们将短暂停Rust程序，转而专注于终端命令。我们先着手解决这个概念，实际实现留待以后再说。  
现在，只要在代码中你觉得屏幕需要刷新的地方，就把这行插入去：

``` rust
print!("\x1b[2J");
```

该 `print! `Rust 中的宏允许我们直接将数据发送到终端。不像 `println!`，结尾为换行， `print!` 只输出我们告诉它的东西。我们从以下开始 `\x1b`，转义字符，或小数点27。  
我们来分析一下 `\x1b`:

- `\x` 表示以下应被解读为十六进制数。
- `1b` 小数点相当于27（你可以查看换算 <a href="https://www.rapidtables.com/convert/number/hex-to-decimal.html" class="external-link" target="_blank">给你</a>).

⠀剩余部分， `[2J`，构成一个 **逃脱序列**.如果你还记得我们之前关于逃脱程序的讨论，比如方向键的逃生，你应该不会惊讶它们还能指示终端执行各种操作——比如清屏。

这里，我们使用了 J 命令 （<a href="http://vt100.net/docs/vt100-ug/chapter3.html#ED" class="external-link" target="_blank">展示中抹除</a>）来擦拭屏幕。2 中的 \[2J“ 告诉终端清除整个屏幕。不同的论点会调整范围：

- `\x1b[1J` 光标清晰。
- `\x1b[0J` 从光标清除到屏幕末端。
- 简单 `\x1b[J` 默认是从光标清除到结束。

想更深入了解终端命令的魔力，可以探索 <a href="https://en.wikipedia.org/wiki/VT100" class="external-link" target="_blank">VT100</a> 现代终端广泛支持的转义序列。详细信息见 <a href="http://vt100.net/docs/vt100-ug/chapter3.html" class="external-link" target="_blank">VT100 用户指南</a>.

看看 <a href="https://learn.microsoft.com/en-us/windows/console/console-virtual-terminal-sequences" class="external-link" target="_blank">关于虚拟终端的文档</a> 并尝试设计自己的逃脱序列，并用上面的命令写出来。

虽然 `crossterm` 这基本上让我们不用手动输入这些序列，理解它们其实很有趣。不过，今后我们就不需要自己写了。

现在让我们的代码更简洁、更有结构：

- 最初，我们启用RAW模式并清空屏幕，设置编辑器。
- 最后，我们进行清理：关闭RAW模式，再次清屏，处理所有错误，向用户告别。
- 在这期间，我们的循环会读取输入、评估、更新屏幕并重复——为未来的自己保持整洁和易于管理。趁代码还能管理，最好现在就开始清理。  
  从现在起，我们不再重复每一次按键，保持屏幕清晰且聚焦。

<a href="https://github.com/pflenker/hecto-tutorial/commit/ba50855df0d40a34ce373db95027ef6e213b8283" class="external-link" target="_blank">看看这次更新的实际操作。</a>  
尝试一下结果，你会发现屏幕正常清空，按键回声消失，退出也很流畅。我们离将 hecto 转变为一个功能齐全的文本编辑器又近了一步！

最后，记住每次清空屏幕可能会遮挡编译器提示。要发现任何警告，就快跑 `cargo build `分开。请记住，Rust不会重新编译未修改的代码，所以要获得新的警告，建议运行` cargo clean` 先，然后 `cargo build`.

在引擎盖下， `crossterm` 使用了逃脱序列 `\x1b[H`， `H` 命令（<a href="http://vt100.net/docs/vt100-ug/chapter3.html#CUP" class="external-link" target="_blank">光标位置</a>），用于定位光标。该 `H` 命令实际上会接受两个参数：行号和列号，用于放置光标。所以如果你的终端尺寸是80×24，并且你希望光标在屏幕中央，可以用命令 `\x1b[12;40H.` （多个参数之间用 a 分隔 `;` 性格。）默认参数 `H` 两者都是1，因此我们可以省略这两个参数，它会将光标定位在第一行第一列，就像我们发送了 `\x1b[1;1H` 指挥部。（行和列编号从1开始，而不是0—— `crossterm` 不遵循此规则，且以0作为起始）。

## Rust中的类型

我们开始聊聊吧 *类型* 现在。直观上， *类型* 是我们可以用来更好地推理我们所讨论的事物的。在第二章我们看到 *类型* 内存中的数据点决定了 Rust 是否处理 `01100001` 作为整数， `97` 或一个角色， `a`.  
因此，我们遇到的第一个类型是 `Char` ，代表一个字符。你可能还记得，即使是这种看似无害、基本的类型，其实已经很模糊了，因为我们对角色的直觉理解和Rust的理解不同（比如我在学校从没学过怎么写回程！）。在第四章中，我们将更详细地探讨字符，并发现Rust中字符实际上是UTF-32编码的。

直到这里，我们遇到过几种类型，但从未真正评论过：

- `Char`如上所述。
- `Editor` 以及 `Terminal` 是我们自己定义的类型。
- A `Result` 是一个类型，表示标记为 的粉色框 `Ok`或者黑盒子，上面写着 `Err`.
- `std::io::Error` 是输入或输出操作期间可能发生的错误的特定错误类型——在我们的情况下，是在从终端读取或写入终端时。
- 然后我们也见过各种所谓的枚举。枚举是一份列出某物可能拥有的不同变体的列表。 `crossterm` 的 `KeyCode` 是一个枚举，表示各种可以按下的潜在键。

编程语言在类型方面有多严格，有一个光谱。在光谱的一端，有以下语言 `JavaScript`，这些根本不在乎类型。如果你想调用 方法 `foobar()` 它不会抱怨，只有在进入那个代码时才检查这个“任何”是否还有叫做 `foobar()`.这被称为“弱类型”。

而在另一端，你有像这样的语言 `Java`这些要求你非常有意识地选择类型，并尽量避免对无效类型调用方法。这被称为“强类型”。

还有一个中间地带，比如一些奇怪的现象 `TypeScript`，这使强类型化到 `JavaScript`，或者 `Groovy`，这使弱类型映射为 `Java`.<sup><a href="#fn3" id="fnref3">[3]</a></sup>

Rust属于“强型”那一类。但如果这是真的，为什么我们要等到第三章最后三分之一才完全遇到他们？例如，为什么我们被允许在我们的 `main()` 刚才？

``` rust
 let editor = Editor::default(); 
```

没有任何关于 `editor`，但 这个 编译。

Rust 在这方面对开发者非常包容。它能推断类型，因为不可能 `Editor::default()` 返回的不是 `Editor`，我们不需要在这个作业中明确说明。作为开发者，我们写的内容要少一些冗长，以及是否应该更改 `default()`，这次更新我们就少了一块代码需要接触。我们可以（有时也必须）明确表达这种类型。如果我们想明确化上述赋值，可以写成：

``` rust
let editor: Editor = Editor::default(); 
```

这样，如果我们改变返回类型 `default()`编译器要求我们也更改 的类型 `editor`.

这被称为 *类型推断*Rust 之所以能做到这一点，是因为我们需要明确在重要位置提及类型，即在 *函数签名*.例如，我们用来移动鼠标光标的函数签名如下：

``` rust
    pub fn move_cursor_to(x: u16, y: u16) -> Result<(), std::io::Error> 
```

内容如下：

- `pub`以下内容可在文件外访问。
- `fn`： 以下是一个函数。
- `move_cursor_to`：该函数以下称为 `move_cursor_to` .
  - `(`： 以下是该函数在调用时所期望输入的所有内容的定义。
    - `x: u16`：该函数首先期望的类型是 `u16`.此后将称为 `x`.
    - `y: u16`：该函数期望的第二种类型必须是 `u16`.此后将称为 `y`.
  - `)`：此时该函数的输入定义结束。
- `->` 以下是该函数将返回给调用者的描述。
- `Result`：该函数返回 `Result`，这是两个可能标记的框之一 `Ok` 或 `Err`.
  - `<`： 以下是该函数返回的两个框的定义。
    - `()`：一个 `Ok` 盒子里什么都没有，会是空的。
    - `std::io::Error`：一个 `Err` 盒子将包含 `std::io::Error`.
  - `>`：这就结束了盒子的定义。

## 整数类型

让我们来看看新类型， `u16`，我们已经见过 `crossterm` 使用并被用于我们自己的代码中。这是表示数字的多种类型之一。正如你可能从隐藏的复杂性中猜到的 `Char`（我们还没看到结局，第四章会详细说），数字也不容易处理。首先，我们需要将整数与其他数字（如分数）区分开来。记住，存储中所有东西都是0和1，所以如果你处理的是分数，你必须思考如何在内存中实际表示它们，这并不简单。  
幸运的是，我们面对的是整数。它们比分馏更容易处理，但仍然有多种口味。先在Rust Playground见见他们，这样你可以试试它们。

``` rust

fn main() {

    let small_u: u8 = std::u8::MAX;  
    let medium_u: u16 = std::u16::MAX;  
    let large_u: u32 = std::u32::MAX;  
    let extra_large_u: u64 = std::u64::MAX;  
    
    let small_i_min: i8 = std::i8::MIN;  
    let small_i_max: i8 = std::i8::MAX;  
    let medium_i_min: i16 = std::i16::MIN;  
    let medium_i_max: i16 = std::i16::MAX;  
    let large_i_min: i32 = std::i32::MIN;  
    let large_i_max: i32 = std::i32::MAX;  
    let extra_large_i_min: i64 = std::i64::MIN;  
    let extra_large_i_max: i64 = std::i64::MAX;  

    println!("Unsigned integers:");
    println!("u8 max: {}", small_u);
    println!("u16 max: {}", medium_u);
    println!("u32 max: {}", large_u);
    println!("u64 max: {}", extra_large_u);

    println!("Signed integers:");
    println!("i8 min: {}, max: {}", small_i_min, small_i_max);
    println!("i16 min: {}, max: {}", medium_i_min, medium_i_max);
    println!("i32 min: {}, max: {}", large_i_min, large_i_max);
    println!("i64 min: {}, max: {}", extra_large_i_min, extra_large_i_max);
}
```

<a href="https://play.rust-lang.org/?version=stable&amp;mode=debug&amp;edition=2021&amp;gist=d9ab1c4613e62957ed2a759c48c2d892" class="external-link" target="_blank">与Rust Playground的联系</a>

我们先从8、16、32和64之间的区别说起：宽度。

这里的宽度是指用多少位来表示这个数字。这个宽度决定了我能表示的不同组合数量。一个假设的2位整数具有以下组合： `00`, `01`, `10`以及 `11`.我可以在那个小整数里存储4个不同的值。该 `16` 在 `u16` 意味着宽度为16位，这给出了更令人印象深刻的65,536个不同值。Rust甚至有宽达128位的整数，导致3.4亿000万亿000万千万000万千万千万种不同的组合。这介于一个十亿兆和一个十二兆响之间。

那么，为什么不谨慎一点，始终使用最宽的整数类型呢？答案是：性能。小批量的机器处理速度更快。32位机器之所以这样称呼，是因为它能够高效地处理32位操作。任何更大的数据类型都必须在底层处理不同，导致性能下降。因此，有时你可能想使用机器上最宽的类型——例如32位机器用u32,64位机器用u64。这就是另一种类型所代表的： `usize` 以及 `isize`，分别。稍后我们会讨论并使用这些。

整数类型的另一种区分用字母i表示，表示（带符号）整数，u表示无符号整数。该 *标志* 这指的是它前面的负数，即整数是否允许为负数。让我们回到我那个超棒的2位整数。正如我告诉你的，它可以代表4种不同的数值，但具体是哪种呢？如果我决定 *带符号整数* ，它代表-2、-1、0和1（按惯例，负值范围稍宽）。如果我决定选择_unsigned integer_s，我可以代表0、1、2和3。<sup><a href="#fn4" id="fnref4">[4]</a></sup>

这意味着在我不预期负数的情况下，无符号数据类型大约能得到两倍数量。

`crossterm` 使用类型 `u16` 放置鼠标，也就是无符号的16位整数。“无签名”很合理——坐标 `(0,0)` 位于终端左上角，因此我们永远不会期望有负坐标。16位，无符号，意味着我们能表示的最高数是65,535。所以我们可以通过接管来推断 `u16` 来自 `crossterm`, `hecto` 在更大的端子上运行不畅。我们暂时可以接受这个。

## 更好的蒂尔兹绘图

接下来的任务是一些底层改进。首先，我对我们写给终端的方式感到不满。我们用了 `println!` 宏是过去的，后来切换到 `print!` 在前一步。但我们也在使用 `crossterm`，我们使用一个宏，称为 `execute!`.这有三个糟糕的原因：

第一——让我不爽的是我们混淆了两个概念——直印和 `crossterm`.对我来说，这种烦躁就像你进房间时闻到的气味。这通常被称为 <a href="https://en.wikipedia.org/wiki/Code_smell" class="external-link" target="_blank">代码气味</a>：这可能意味着更深层的问题。

第二——我们不理解 `execute!` 然而，我们只是因为文档要求才用它。我觉得这也有点蹊跷。

第三——打印有时候很奇怪，我们的代码没考虑到这一点。

先从最后一点说起，印刷有多奇怪。印刷到屏幕是一项昂贵的操作，为了提高效率，存在缓冲区的概念。每当你尝试在屏幕上写入时，实际上你是写入这个缓冲区，而这个缓冲区会被清空（并写入屏幕），无论是手动写还是缓冲区满了。我们之前还没遇到这个问题，因为缓冲区通常会 *线路* 缓冲区，因此如果缓冲区内有整行，最迟会自动写出。到目前为止，我们所有的打印都包含了新行，所以没遇到这个问题。但写转义序列不一定会打印出新行，这意味着那时我们可能会遇到缓冲问题，

这就引出了第二点，我们并不了解自己在用什么。 `execute!` 实际上是一个宏，确保你传递给它的任何内容都能立即打印出来。那是......半好？这几乎是我们想要的，但正如我上面提到的，写入是一项昂贵的操作，我们真正需要在评估完按键操作后才完全写入屏幕。我们不需要每次光标放置都写完整，清除屏幕或写入，可以让缓冲机制决定，系统也可以自行判断写入的最佳时机。如果我们一直写出每一个细微变化，甚至可能导致奇怪的闪烁效果。

这就引出了第一个问题，代码气味。调查“代码气味”让我们开始阅读相关资料 `execute!` 并揭示印刷技术的奇特。结果是，我们现在明白，我们的代码之所以能以某种方式工作，纯属偶然，而且它运行效率并不高。

## 移动语义与所有权

突击测验！以下结果是什么：

``` rust

fn main() {
    let x = Hecto{};

}
```

（忽略 `derive` 暂时来说，这对理解这里来说并不是关键）

<a href="https://play.rust-lang.org/?version=stable&amp;mode=debug&amp;edition=2021&amp;gist=74136d711e93c1c4caf4d919ba369798" class="external-link" target="_blank">试试Rust Playground吧</a>.

答案是：没用，它行不通。听起来怪怪的？  
那再来一场快问答怎么样！

以下结果是什么：

- 我把一个苹果放在你左手里
- 你咬了一口苹果
- 我把这个苹果移到你右手边
- 你用左手咬苹果，然后用右手咬苹果。

答案是：没用，它行不通。让我们学习吧。

Rust非常非常努力地确保记忆安全。让我们用以下比喻来说明这个问题：

如果你是拥有大量财富的自豪者，携带它并不总是实际，因为它很重。你把它放在某个地方，带着一张地图，或者多张指向你财富不同部分的地图。那些地图代表你的财富。<sup><a href="#fn5" id="fnref5">[5]</a></sup>

如果你自豪地拥有一个庞大的数据结构（比如文件），移动内存中的文件并不总是实际的，因为它很重。你把它放到某个地方，带着一个指针（我们称之为 *去引用*).那个指针代表你的文件。

如果你把一张地图交给我，我不仅能访问你藏在那里的钱，还能访问你藏在别处的钱，那么你以为指向钱的地图突然指向我提供的欠条。同样，如果你复制地图，把一张交给我，一张交给你的朋友，然后我取出钱，你的朋友会在预期中发现一张欠条。  
你的藏身处不安全。

如果你把你文件的一个指针交给我，我可以删除你另一个文件，那么你的其他指针突然指向别的东西。同样地，如果你把指针交给我和其他函数，我可以更改你的文件，那么另一个函数会发现一个它可能没预料到的修改文件。  
你的记忆不安全。

不同的编程语言使用不同的方法来解决这个问题。 `C` 根本解决不了，把它留给你这个开发者。 `JavaScript` 一举成真。让我们了解一下Rust是如何做到这一点的。

Rust 对跟踪非常严格 *谁* 确实 *什么* 每个值。我们上面看到的是“移动语义”：Rust 将所有权转移到 `struct` 实例来自 `x` 到 `y`， 尝试通过以下方式访问 `x` 此时已无法实现。

为什么Rust不干脆 *收到* 值，使得 `x` 以及 `y` 是分开的吗？  
Rust 会在合适的时候这样做。试试上面的例子，不是用结构，而是用整数 `x` 以及 `y`你会发现它完全正常工作。  
但对于更复杂的数据类型，例如 `struct`，Rust其实并不太会做。Rust 上面提出的建议是，我们传递的不是真实的东西，而是指向它的指针，或者告诉 Rust 它如何复制那个东西。它 *可以* 猜猜看，正如我们稍后会看到的，但我们先完全理解“特质”的概念。

## 特征

我们现在面临这样一个局面：为了实现某些目标，在这里是复制某些东西，我们需要实现一些东西。

我们已经在这里那里见过类似的情况。请考虑上面示例中的这条线：

``` rust
println!("x is {:?}",x); 
```

这只有在Rust知道如何创建 `Debug` 表示 `x`.Rust 知道如何处理像这样简单的数据类型 `u16`，但不包括 `struct`s.

所以Rust需要一个方式告诉我们：嘿，按顺序印刷 *这是调试时的做法*，我希望你去做 *告诉我怎么做*!  
我们需要一种方式告诉Rust： *这就是正确的做法！*

我们要找的就是所谓的“特质”。A *特征* 就是这样：一句话，说“你需要实现几个功能才能实现”。

让我们稍微写点代码，深入探讨这个问题。假设我们想要新的、闪亮的测试结构， `hecto`，显示为调试字符串。我们试试这个：

``` rust
struct Hecto;
fn main() {
    let x = Hecto;
    println!("{:?}",x);
}
```

<a href="https://play.rust-lang.org/?version=stable&amp;mode=debug&amp;edition=2021&amp;gist=fdb6fdf484af983f6a984f780dcf33f7" class="external-link" target="_blank">看看Rust Playground上的这个。</a>

这不会编译，编译器会特别说明原因：

      = help: the trait `Debug` is not implemented for `Hecto`

好的，理解了，“该特性未被实现”是指类似“我们必须编写代码才能实现”之类的意思。 <a href="https://doc.rust-lang.org/std/fmt/trait.Debug.html" class="external-link" target="_blank">医生们</a> 告诉我们这个特征是什么样子：

``` rust
pub trait Debug {
    fn fmt(&self, f: &mut Formatter<'_>) -> Result<(), Error>;
}
```

这说明：这是一个被称为 `Debug`为了实现该特征，你需要实现一个名为 `fmt` 用那个特定的签名。

如果你试着聪明一点，稍微改一下打印语句，改成：

``` rust
println!("x is {}",x); 
```

然后这个方法也会失败，这次是因为 Rust 希望你实现 `Display` 特质。

根据 <a href="https://doc.rust-lang.org/std/fmt/trait.Display.html" class="external-link" target="_blank">纪录片</a>，该特征如下所示：

``` rust
pub trait Display {
    fn fmt(&self, f: &mut Formatter<'_>) -> Result<(), Error>;
}
```

所以这基本上是完全相同的函数签名。这意味着我们不能简单地实现这个函数，我们必须以一种让 Rust 理解哪个函数实现哪个特征的方式来实现，这样实际的调试和显示表示就不同了。

好吧，那我们就这么做，但我们会有点孩子气。我稍微作弊一下，直接说说如何处理格式化器，因为这里的重点不是格式化器的工作原理，而是：我们如何告诉 Rust 我们想实现这个方法？

答案是我们用特殊方式实现了该函数 `impl` 方块，看起来像这样：

``` rust
impl Debug for Hecto {
// Implementation goes here
}
```

这意味着：“本实现块实现了 `Debug` 数据类型的特征 `hecto`.

我们试试这个。

``` rust
use std::fmt::{Debug, Result, Formatter};
struct Hecto;
impl Debug for Hecto {
    fn fmt(&self, f: &mut Formatter<'_>) -> Result {
        write!(f, "Hecto is awesome.")
    }
}
fn main() {
    let x = Hecto;
    println!("{:?}",x);
}
```

我们在这里要说的是：随便 `hecto` 无论它包含什么，你唯一需要的调试信息就是“hecto 很棒”。

果然：这次代码编译成功，调试信息正是我们实现的。

## 衍生与利用特征

我们现在正逐渐从上方理解这两个问题。

`Stdout` 不直接实现 `flush()`.相反，它实现了一个名为 <a href="https://doc.rust-lang.org/std/io/trait.Write.html" class="external-link" target="_blank">写</a>，且该特征要求 `flush()`.原因是在代码的某些节点，我们可能不想说“我们期望 a `Stdout`这里。”我们可能想说“我们不在乎这里是什么，只要它实现了 `Write` 特质“。隐含地，这就是 `println!` 确实如此。它没有说“我想买一个 `hecto` 这里。”它不认识这个东西。所以它会说“我想在这里有实现调试功能的东西”。

代码行 `use std::io::Write;` 将该特征纳入范围及其实现，并使 `flush()` 我们可以使用。你完全可以自己写实现 `Write`，包括你自己的方法 `flush()`，然后用这个代替这里。

好了，这就解释清楚了。那另一个呢？

我跟你说过，Rust 更上去了 *可以* 帮我们猜一些特征，但我们必须明确告诉它，以避免自动实现各种我们永远不会用的特征。就是这样 `derive` 确实如此。

如果我们不是为 hecto 实现调试特性，而是派生它，那么我们之前的代码示例会是这样的：

``` rust
#[derive(Debug)]
struct Hecto;
fn main() {
    let x = Hecto;
    println!("{:?}",x);
}
```

<a href="https://play.rust-lang.org/?version=stable&amp;mode=debug&amp;edition=2021&amp;gist=aaa47f47dd01888e94d484a8493b771c" class="external-link" target="_blank">看看Rust Playground上的效果。</a>

我们也可以让Rust推导出复制物品所需的特征（`Copy` 以及 `Clone`).让我们把之前的突击测验延伸到这里：

``` rust

struct Hecto;
fn main() {
    let x = Hecto{};
    println!("x is {:?}",x); // This would prints out "x is Hecto"
    let y = x;
    println!("x is {:?}, and y is {:?}",x, y); // ... and this?
}
```

当然，这正是奏效的。

那么，怎么做到的 `Clone` 偷偷溜进来？这现在不是特别重要，但既然你问了： `Copy` 旨在成本低廉、隐含且快速。你自己不能实现这个，Rust需要帮你完成。 `Clone` 它可以很强大，你也可以自己实现。

现在，让我们回到顶部——为什么行为会有差异 `Debug` 以及 `Release`?  
我在第一章已经铺垫过了。原因是：这个漏洞产生的行为非常令人困惑，而且大多数情况下绝对不是你想要的（这在中间两个问题中尤其如此，我们稍后会详细探讨），所以每当这种情况发生时，Rust 会选择最极端的方式告诉你你错了：它崩溃。

不过 Rust *仍然可以* 继续运行，因为它确实得到了一些可供处理的值。这些值非常非常错误，但毕竟还是值。假设对最终用户来说，程序崩溃比得到严重错误的数据更糟，Rust 就会继续运行。

但发生了什么 `huge`以及发生了什么 `small`?  
让我们回到我上面提到的那个小数据结构，2位整数。我们还发明一个3位整数，并称呼无符号的变体为 `u2` 以及 `u3`，分别。他们能存储的物品如下：

<div class="table-wrapper">

| 编号 | `u2` | `u3`  |
|:----:|:----:|:-----:|
|  0   | `00` | `000` |
|  1   | `01` | `001` |
|  2   | `10` | `010` |
|  3   | `11` | `011` |
|  4   |      | `100` |
|  5   |      | `101` |
|  6   |      | `110` |
|  7   |      | `111` |

</div>

如果你还记得早期的章节之一，操作员 `as` 对Rust说：对这个底层数字换个角度。我们用这个方法让 Rust 把二进制数当作 `Char`.  
假设我们这样做 `let x: u2 = 3` .那么在罩盖下，x 被存储为 `11`.如果我们真的这么做 `let y = x as u3`，它只是顺其自然，因为 `011` 对于 `u3` 表示3，表示为3，同理。 `11` 对于 `u2`.

但如果我们真的去了 `let x: u3 = 7` 随后 `let y = x as u2`，我们取一个在罩子下表示为 `111` 然后把它变成 `11`，即：3。7变成3。

事情是这样的 `huge` 变得如此渺小。

让我们看看另一个奇怪现象。再看看上面的表格，怎么说 `u2` 每次递减时都会有变化。从二进制数中去除一的算法似乎是：

1.  从右边开始。
2.  如果当前数值为1，则将其减为0。你完了。
3.  如果当前数字是0，翻转为1，向左移动一步。
4.  去2号。

如果你在0上执行此事，表示为 `00`?算法会将你的数字翻转为 `11`，组成3，然后结束。0-1是3。这被称为 *下溢*.反过来也有——它叫做 *溢出*.

事情是这样的 `small` 变得如此庞大。

你可能会以为那次坠毁完全是因为那个地方我们试图为波浪号腾出空间却没检查 `padding`.是的，确实如此，但问题的一部分也在于我们当时没有系安全带。
