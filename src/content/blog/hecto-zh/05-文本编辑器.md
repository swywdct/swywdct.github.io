## 作业15：插入普通角色

我们先专注于在文本中添加普通字符。由于我们无法确定该字符是否会触发新字素簇的形成，因此需要重建内部结构 `Line` 插入时。

这里需要注意的一点是 *运动*：用户认为的插入其实是一个插入，然后向右迈步——否则我们会倒着输入。另一个需要考虑的方面当然是字素和字素簇。插入的字符完全有可能成为已有的字素簇的一部分——在这种情况下，任何额外的移动都会导致插入符号向右移动过远。或者，新加入的角色可以让左右字形合并——想象在成人表情和零宽度连接者之间添加一个握手表情——这样它们就会形成“成人牵手表情”，实际上 *移除* 字素。

我们应对这个话题的策略是检查编辑后字素数量是否发生变化：如果增加，我们就移动，否则保持原位。除了插入新角色后成长之外，我们无法知道应该向任何方向移动多远。这应该涵盖了我们所针对的案例 `hecto`.

这是作业内容：

## Rust的借用检查器实际操作

我们在之前的章节中已经见过Rust对类型安全的处理方式。在修改数据方面，Rust 对可变和不可变引用的处理有严格的规则。

请考虑我提交中的这个函数：

``` rust
  pub fn delete(&mut self, at: Location) {
        if let Some(line) = self.lines.get(at.line_index) {
            if at.grapheme_index >= line.grapheme_count()
                && self.lines.len() > at.line_index.saturating_add(1)
            {
                let next_line = self.lines.remove(at.line_index.saturating_add(1));
                self.lines[at.line_index].append(&next_line);
            } else if at.grapheme_index < line.grapheme_count() {
                self.lines[at.line_index].delete(at.grapheme_index);
            }
        }
    }
```

该函数访问当前行两次：一次在周围 `if let Some(line)`，随后在调用时 `append` 或 `delete`.无法重构这段代码来使用。 `get_mut` 周边地区 `if let` 然后取消那个额外的访问权限。为什么？

### 内存安全与可变引用

Rust通过强制对可变数据的独占访问来确保内存安全。该 `&mut self` 函数签名意味着在 delete 执行时，程序的其他部分无法修改自结构（`Line` 在这里）。这就像借用实物一样：只要你拥有它，别人就不能使用它。  
排除首字母 `if let`注意，self.lines（因此self）在对其中元素的任何引用时并未发生变化 `self.lines` 活跃。移除 `next_line` 是一个不同的操作，对 `self`，接着是呼叫 `append`.

### 为什么没有借阅检查器错误

那么问题是：为什么这个参考没有 `line`，源自 `if let Some(line)`，触发借款检查器错误？答案在于其使用的时间和范围。它只用过一次，在任何变异行动之前。之后，该引用即无效。在可变操作开始后添加任何在线操作，即使是简单的读取，也会使代码失效并导致编译器错误。

## 特征与自动实施

我们还来讨论一下我提交中的另一个代码片段：

``` rust
impl fmt::Display for Line {
    fn fmt(&self, formatter: &mut fmt::Formatter) -> fmt::Result {
        let result: String = self
            .fragments
            .iter()
            .map(|fragment| fragment.grapheme.clone())
            .collect();
        write!(formatter, "{result}")
    }
}
```

此阶段的目标是实现 `to_string` 函数转动 `Line` 变成了 `String`用于附加一 `Line` 对另一个人来说。那我为什么要实现 `Display` 这里？

在我解释之前，我想强调，不像我那样做是完全合理的做法。我走这条路是为了突出一些关于Rust的有趣之处——只是实现一个简单的 `to_string` 没问题。

我在这里实现的是 `Display` 特质。你可能还记得之前章节中提到，这个特性允许将数据结构打印给用户，例如： `println!(“{my_struct}”);`.将某物转换为字符串是一个非常常见的操作，因此存在一个特征： <a href="https://doc.rust-lang.org/std/string/trait.ToString.html" class="external-link" target="_blank">ToString</a>.还有一个对应的特质，我们也说 *可以* 已实现： <a href="https://doc.rust-lang.org/std/str/trait.FromStr.html" class="external-link" target="_blank">FromStr</a>.正如你现在已经知道的，在某些情境下实施特质会带来好处。但我在这里想指出的，以及我为什么要实施 `Display`，且 `ToString`，是实现某些特征会自动实现另一个特征。我们在实现 `From` 特质并实现了 `into` 免费——而且在这个案例中，我们正在实现 `Display`，导致自动实现 `ToString` 特征，该特征提供 `to_string`. `Display` 通常被认为比实现更为灵活 `ToString`.

其基本概念称为a <a href="https://doc.rust-lang.org/book/ch10-02-traits.html" class="external-link" target="_blank"><em>全面实现</em></a>： 你可以为任何实现特征B的类型写一个通用的特征A实现，实际上使特征B也实现了特征A。

## 作业18：Tab和Enter。

我们还需要支持另外两个关键： `Tab`，只需插入一个 `\t`， 和 `Enter`.

要处理 `Enter`，我们需要扩展 `Line` 以一种允许我们在当前位置分割线路的方式——缩短当前线路，返回新的线路 `Line` 剩下的。除了使用 `push()` 在 a 的结尾添加一个条目 `Vec`，我们可以使用 `insert(index: usize)` 在目标位置添加物品。后续条目将向右移动。`Vec` 也允许通过呼叫来分开 `split_off`.

这里有一个简短的Rust Playground例子来说明：

``` rust
pub fn main() {
    let mut first_vec = vec!["Entry #1".to_string(),"Entry #2".to_string()];
    println!("{first_vec:#?}");
    
    for num in 4..=10 {
        first_vec.push(format!("Entry #{num}"));

    }
    println!("{first_vec:#?}");
    // Oops, missed something!
    first_vec.insert(2, "Entry #3".to_string());
    println!("{first_vec:#?}");
    
    let second_vec = first_vec.split_off(5);
    println!("{first_vec:#?}");
    println!("{second_vec:#?}");
    
}
```

## 文本编辑器中的数据结构

我们现在已经建立了 `hecto`.去除外部结构体后，我们会为每个可见字素存储元数据 `vec`.这些 `vecs`，每个代表一条直线，然后被归入另一条线中 `vec`.这种配置在处理格式化文本的文本编辑器中相当常见。例如， <a href="https://www.slatejs.org/examples/richtext" class="external-link" target="_blank">SlateJS</a>，一个基于JavaScript的富文本编辑器，利用节点捕捉应用于文本段的格式属性，如下所示：

``` json
[
      {
        "text": "Hello,",
        "bold": true
      },
      {
        "text": " World",
        "bold": true,
        "underline": true
      },
      {
        "text": "!",
        "underline": true
      }
]
```

正如你所想，寻找和定义适合文本编辑器的数据结构现在已经是一个广为人知的问题，这包括对性能的推理、插入类型（例如插入通常发生在同一区域，还是随机分布在文本中？）以及其他考虑因素。文本中常见的数据结构是 <a href="https://en.wikipedia.org/wiki/Rope_%28data_structure%29" class="external-link" target="_blank">绳索</a>.绳索将文本拆分成更小、易于管理的部分，并将它们组织成二叉树结构。

绳索的魅力在于它的高效性——尤其是在你需要插入或删除文本时。有了Rope，这些操作不需要移动大量文本块，即使文档变长也能保持顺畅。使用绳索导航也更高效，因为树中的每个节点都会记录其下方的文本长度，方便快速跳转到文档的不同部分。

在这个教程中，我们会坚持使用效率较低的内部数据结构，但如果你有兴趣学习如何在 Rust 中使用绳索，可以看看 <a href="https://docs.rs/ropey/latest/ropey/" class="external-link" target="_blank">绳索</a>.

- 我们保存文件的方式，比如 `writeln!`也为我们正确处理换行。

## 用户界面

我们的文本查看器目前无法向用户显示除文本本身以外的任何信息。因此：

- 我们无法告知用户文件打开或保存失败
- 我们无法告知用户存档是否成功
- 用户需要猜测如何保存文件
- 我们无法为用户提供保存新文件的方法，因为没有任何输入提示让用户添加自己的文件名。

这样不行，所以我们会先构建一个能向用户展示有用信息的用户界面，然后再调整它来处理提示。更具体地说，我们将在编辑器底部添加两行：A *状态栏*，该信息将包含关于该文档的信息，以及 *消息栏*，该系统会向用户显示通知。稍后，我们还会使用消息栏来允许用户输入。

- 如果终端高度低于3行，则仅显示状态栏和消息栏。

就是这样 `checked_sub` 做：它尝试执行减法。如果成功，则返回结果。如果没有，在下溢时返回为 N。

我们现在希望消息最多5秒后消失。为此，我们会跟踪消息首次渲染的时间，每次重新渲染时，检查是否已经过5秒或更长时间。如果是这样，我们会隐藏这条信息。

为此，我们将使用 <a href="https://doc.rust-lang.org/std/time/struct.Duration.html#method.new" class="external-link" target="_blank">持续时间</a> 以及 <a href="https://doc.rust-lang.org/std/time/struct.Instant.html" class="external-link" target="_blank">瞬间</a>.让我们在这个Rust Playground见见他们：

``` rust
use std::time::Duration;
use std::time::Instant;
use std::thread;


fn main() {

    let before_sleep = Instant::now();
    let five_seconds = Duration::new(5, 0);
    let seven_seconds = Duration::new(7, 0);
    let ten_seconds = Duration::new(10, 0);

    println!("Going to sleep now for seven seconds!");
    thread::sleep(seven_seconds); // Sleep for more than 5, but less than 10 seconds
    let after_sleep = Instant::now();
    
    if after_sleep - before_sleep > five_seconds {
        println!("Five seconds have passed!");
    }

    if after_sleep - before_sleep > ten_seconds {
        println!("Ten seconds have passed!");
    } 
    
    let elapsed = after_sleep.duration_since(before_sleep); // Calculates the elapsed time
    println!("Actual sleep time: {} seconds and {} nanoseconds", elapsed.as_secs(), elapsed.subsec_nanos());

}
```

## 作业25：代码复习

<a href="https://github.com/pflenker/hecto-tutorial/commit/28658f8d43a9d51ff088502235dff1438919f5cb" class="external-link" target="_blank">这是我的代码。</a>

在结束本章之前，让我们仔细看看我做的一个代码修改，因为它帮助我们更好地理解一些设计和Rust的概念。  
是关于我的 `FileInfo` 结构。

以下是更新后的代码，注释中提到了相关变更：

``` rust
#[derive(Default, Debug)]
pub struct FileInfo {
    path: Option<PathBuf>, // No longer pub
}

impl FileInfo {
    pub fn from(file_name: &str) -> Self {
        Self {
            path: Somefrom(file_name),
        }
    }
    // New fn:
    pub fn get_path(&self) -> Option<&Path> {
        self.path.as_deref()
    }

    //New fn:
    pub const fn has_path(&self) -> bool {
        self.path.is_some()
    }
}
```

首先，让我们讨论代码更改背后的动机。

在我们引入 `save_as` 功能，从来只有一个 `FileInfo` 可用，实际上是不可变的：一旦设定， `FileInfo` 从未改变，而且它要么是 `PathBuf` （加载文件时）或者没有。

随着 `save_as`，我必须做出决定：是 `FileInfo` 某种可变的——暗示调用 `save_as` 需要 `FileInfo` 以更新其 `path` - 还是说它不可变 - 意味着 `FileInfo` 需要用另一个存档替换吗？

我选择了后者。我觉得这对 `hecto` 现在是想让这个决定百分百正确，但上面的代码更改反映了我想保留的愿望 `FileInfo` 不可变：不允许直接访问 `path`，我创建了一个getter函数，并且还暴露了 `has_path` 作为一种便利功能。

这解释了 *为什么*，但不包括 *怎么会这样* - 上述代码变更带来了两个令人惊讶的效果：

- 它返回一个 `Option<&Path>` 而不是 `Option<&PathBuf>`
- 它叫 `as_deref` 关于 `self.path` 去做这件事。  
  为什么？

首先， `Path` 以及 `PathBuf` 与 非常相似 `String` 以及 `str` 这也是我在上面代码中使用这两个原因的原因。这允许我们将代码转换成我们更熟悉的版本——这也有助于我们更容易理解 `as_deref`.我们通过改变来实现这一点 `path` 在结构中如下：

``` rust
pub struct FileInfo {
    path: Option<String>,
}
```

我们第一次尝试返回 `path` 可能长这样：

``` rust
    pub fn get_path(&self) -> Option<String> {
        self.path
    }
```

这行不通，如果你还记得我们之前讨论过的移动语义，原因就很明显了：如果我们移动 `String` 我们自己无法保持——现有的 `Option` 会被否定。 `String` 不实现 `copy` 特性，所以路径的副本不会被自动返回。

没关系，我们不需要复制或克隆它，因为我们不打算修改它。那这样怎么样？

``` rust
    pub fn get_path(&self) -> Option<&String> {
        self.path.as_ref()
    }
```

这招有效！ `as_ref` 将选项中的物品变成对该物品的引用，这正是我们想要的。与 `&String` 代替 `String` 虽然有点棘手，但这通常是一种有效的方法。但如果我们想退回 `&str` 相反，前提是不需要 的开销 `String`，反正它本来就应该是只读的。这引出了我们的观点：

``` rust
    pub fn get_path(&self) -> Option<&str> {
        self.path.as_deref()
    }
```

`as_deref` 其对立 `as_ref` 上面，如果你还记得去引用 `String` 得到 `&str`，现在很容易理解 `Option<&str>`，如愿。
