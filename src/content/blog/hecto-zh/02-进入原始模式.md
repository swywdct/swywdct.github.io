# 第 2 章：进入原始模式

## 第二章：进入原始模式

<img src="/images/hecto-zh/flenker-1712509029-0.png" alt="flenker-1712509029-0.png|philipp&#39;s blog" />

我们试着读取用户的按键记录。  
把所有东西都移除 `main.rs` 并用以下代码替换：

``` rust
use std::io::{self, Read};

fn main() {
    for b in io::stdin().bytes() {
        let c = b.unwrap() as char;
        println!("{}", c);
    }
}
```

试着玩玩这个程序，看看它是怎么工作的。要阻止它，按压 `Ctrl-C` 或 `Ctrl-D`.

一旦你觉得已经足够理解了这个项目，就可以前往 <a href="https://github.com/pflenker/hecto-tutorial/commit/73c0e2acd8767953b469102b1fd8ff99d8c979c4" class="external-link" target="_blank">此步骤在GitHub上</a> 对代码和语法的深入解释。

如果你猜上面的代码会打印你在新行输入的任何字符，那你就错了。或者说是半对的——程序就是这么做的，但它不会立刻从终端获取字符进行处理。

当你奔跑时 `./hecto`，你的终端连接到标准输入端，因此你的键盘输入会被读取到 `b` 变数。不过默认情况下，你的终端是 **正则模式**，也称为 **烹饪模式**.在此模式下，只有用户按下键盘时，键盘输入才会发送到程序 `Enter`.

这意味着我们的程序允许你输入任何你想要的文本，一旦按下回车，它会输出相同的文本，每个字符都换成一行，然后等待新的输入。如果你按 `CTRL-D`你说 `io::stdin().bytes()` 它应该读取的文件已经结束。 `CTRL-C` 立即终止该项目。

这种行为对许多程序都很有用：它允许用户输入一行文本，使用 `Backspace` 修正错误，直到他们能完全按照自己的意愿获得输入，最终按下去 `Enter` 把它发给项目。这就是终端的工作原理，和Rust没有关系。但对于用户界面更复杂的程序，如文本编辑器，这种方法并不适用。我们希望每一次按键输入时都能处理，以便能立即响应。

## 媒体 `q` 放弃？

为了演示规范模式的工作原理，程序在读取 a 时退出 `q` 用户按键。为此，请在下方添加以下代码 `println` :

``` rust
if c == 'q' {
    break;
}
```

该 `break` 语句将结束 `for-each` 如果读取的字符是 `q`.

要退出该程序，你需要输入包含以下内容的一行文字 `q` 然后按下回车键。程序会快速逐字符读取文本行，直到读取 `q`，此时 `for..in` 循环会停止，程序也会退出。 *也不是* 按下后程序会立即结束吗？ `q` （你必须击中 `Enter` 之后的某个时间）， *也不是* 它会处理之后发生的任何事情吗？ `q` - Rust在离开时丢弃额外角色。

## 进入RAW模式

我们想要的是 **RAW模式**，我们将使用依赖关系来将终端切换到它。<sup><a href="#fn1" id="fnref1">[1]</a></sup> Rust 中的外部依赖称为 Crates（`cargo`，箱子......明白了吗？），我们要用的那个叫做 <a href="https://docs.rs/crossterm/latest/crossterm/" class="external-link" target="_blank">跨项</a>.

我们加 `crossterm` 通过执行以下命令：

    cargo add crossterm

这会产生以下输出：

        Updating crates.io index
          Adding crossterm v0.27.0 to dependencies.
                 Features:
                 + bracketed-paste
                 + events
                 + windows
                 - event-stream
                 - filedescriptor
                 - serde
                 - use-dev-tty
        Updating crates.io index

`cargo` 更新其索引从以下 <a href="https://crates.io" class="external-link" target="_blank">crates.io</a> 然后 加 `crossterm` 对先前空无一物 `dependencies` 该分段 `Cargo.toml`.以下是该依赖被启用（+）或禁用（-）的功能列表。功能内容不仅限于这个教程，但你可以阅读相关内容 <a href="https://doc.rust-lang.org/cargo/reference/features.html" class="external-link" target="_blank">给你</a>.

下次你跑步的时候 `cargo build`你会看到的 `cargo` 将建造远不止 `hecto`，因为它现在也在构建 `crossterm` 以及它所使用的依赖关系。

现在，让我们利用我们闪亮的新依赖关系，替换 的内容 `main.rs` 代码如下：

``` rust
use std::io::{self, Read};
use crossterm::terminal::enable_raw_mode;
use crossterm::terminal::disable_raw_mode;

fn main() {
    enable_raw_mode().unwrap();
    for b in io::stdin().bytes() {
        let c = b.unwrap() as char;
        println!("{}", c);
        if c == 'q' {
            disable_raw_mode().unwrap();
            break;
        }
    }
}
```

试试看，你会发现你输入的每个字符都会立刻被打印出来，而且一输入就立刻完成 `q`节目结束。

## 依赖管理

现在也是更详细讨论依赖管理的好时机，并将这里看到的内容与第一章的一些内容联系起来。

使用依赖意味着我们可以专注于编码任务，而不必一遍又一遍地重新发明轮子（以及同样的bug）。

该 `Cargo.toml` 在其中起着重要作用。我们不仅会在那里列出依赖关系，如果我们自己构建库函数，还可以使用 `Cargo.toml` 以便提供关于我们套餐的更多信息，以备我们决定发布的话。依赖关系之间有其他依赖关系，而这些依赖关系还依赖于其他因素，这是正常且有时令人遗憾的。这意味着我们基本上需要信任库的作者，a） 做个好人，b） 如果作者信任或核实过 *他们的* 依赖关系包括 *还有* 好人，诸如此类。这可能是一个严重的问题。<sup><a href="#fn2" id="fnref2">[2]</a></sup>

此时， `cargo`能够选择性地重新编译已更改代码部分的能力非常有用，因为重新编译整个代码可能需要一些时间。编译时间则不是 *糟糕* 不过——只添加一个依赖后，这就令人担忧了——但新建的区别 `cargo build` 而且之后的一颗已经很明显了。

如果你在 `target` 目录，正如我们在第一章中所做的，你现在也会注意到那个恰如其名的文件夹 `deps` 现已填满。你还会注意到，生成的可执行文件 `release` 构建是 *非常多* 小于得到的可执行文件 `debug` 建造！

我们还试着理解第一章中遗漏的最后一块拼图：那 `Cargo.lock`.该文件还包含了关于依赖关系的信息——甚至比 `Cargo.toml`.但它有什么用？

如上文GitHub步骤所述， `cargo` 愿意升级不含破坏性变更的版本。假设 `crossterm` 引入了修复错误但其他保持不变，这种行为意味着我们只需请求 `cargo` 通过运行更新索引 `cargo update`然后后续构建将使用更新后的版本。你只会注意到更长的构建 `cargo` 下载它。

这带来了问题。假设你已经成功释放 `hecto` 而且它在全球范围内被使用。然后有人提交了bug报告，你试图复现它。你打开你的旧货 `hecto` 代码库，重建它，你找不到错误。  
这意味着以下其中之一：

- 你没有正确复刻它
- 你的机器和另一台不一样，这就是错误的来源
- 这个bug是因为依赖系统有缺陷，但这个依赖修复了bug， `cargo` 下载了，意思是你的新版本没有这个功能

这真的很烦人。该 `Cargo.lock` 它试图通过提供一个非常具体的“指纹”来解决这些问题，涵盖你构建中所有依赖。你可以基于依赖关系构建 `Cargo.lock` 通过使用：

    $ cargo build --locked

`cargo` 然后检查 `Cargo.lock` 仍然符合 `Cargo.toml` （因此，定义一个新的依赖或版本 `.toml`这不属于 `.lock` 但又会引发错误），但它并没有使用与之前使用的版本不同的版本，即使期间有更新。

## 进入RAW模式，艰难的

## 显示按键

为了更好地了解RAW模式下的输入是如何工作的，我们来打印每个读取的字节。我们会打印每个字符的二进制表示、数值ASCII值，以及如果是可打印字符所代表的字符。我们通过改变 `main()` 具体如下：

``` rust
fn main() {
    enable_raw_mode().unwrap();
    for b in io::stdin().bytes() {
        let b = b.unwrap();
        let c = b as char;
        if c.is_control() {
            println!("Binary: {0:08b} ASCII: {0:#03} \r", b);
        } else {
            println!("Binary: {0:08b} ASCII: {0:#03} Character: {1:#?}\r", b, c);
        }
        if c == 'q' {
            disable_raw_mode().unwrap();
            break;
        }
    }
}
```

前往 <a href="https://github.com/pflenker/hecto-tutorial/commit/8121519e5cb32b7a49fa3f8a80720daa0ed28f6d" class="external-link" target="_blank">此步骤在GitHub上</a> 想多了解一些代码细节。

在我们一起尝试之前，先让我快速告诉你一些关于在终端打印物品的有趣事实。但首先：打字机。

<a href="https://en.wikipedia.org/wiki/Typewriter" class="external-link" target="_blank">打字机</a> 工作原理是这样的：一张纸被固定在鼓上，鼓又被固定在一个车厢上。每当你敲击带有字符的键时，机械印章机构会将对应字符印在纸上，并使滑车向左移动。当你快读完论文时，会听到一声“叮”的声音，告诉你现在应该在意分段，因为论文快结束了。然后你需要向右推一个把手。那个手柄有两个作用：它移动滑车，让你能完全从左边开始新一行，同时转动鼓，避免覆盖上一行，而是重新开始新的。

这三个概念——铃声、鼓的转动和马车的移动——仍然存在于计算机中，尽管历史上这些概念是通过 <a href="https://en.wikipedia.org/wiki/Teleprinter" class="external-link" target="_blank">电传打字机</a> 给电脑。<sup><a href="#fn3" id="fnref3">[3]</a></sup> 回到 `hecto`你可能已经注意到了 `println!` *确实如此* 开一条新线，但它并没有启动它 *左侧*就像打字机上我们只旋转鼓，但不移动滑架一样。我们通过添加回车信号（实际上仍是这样称呼）来解决这个问题， `\r`，到上述输出。

顺便说一句，我们在代码里对想让 Rust 处理哪些部分（比如换行，是 Rust 添加的）有些犹豫不决 `println!`），以及我们想自己做的事情（比如回车）。我们以后再说。

现在，让我们用新程序学习几件事。

在底层，字符只是数字，数字在内存中表示为 `0`s 和 `1`（我们称这些为“比特”）内部。假设我们有 `01100001` 在我们的内存中（我们称8位为a） *字节*，所以这是一个字节）。是 <a href="https://www.rapidtables.com/convert/number/binary-to-decimal.html" class="external-link" target="_blank">将二进制数字转换为十进制数的简单方法</a>，如果我们这样做，则 `01100001`，我们得到这个数字 `97`.存在一种标准，将这样的数字映射到一个字符。这个标准叫ASCII，当你 <a href="https://www.asciitable.com/" class="external-link" target="_blank">在ASCII表格中查找97</a>，结果是小写 `a`.果然，如果我们输入 `a` 在我们的新节目中，我们看到这三样东西都被打印出来： `a`, `097` 以及 `01100001`.  
这也是为什么我们可以轻易地教程序把从键盘上读取的二进制视为字符。基本上我们只是告诉电脑“请把这个数字当作一个角色来处理”。底层的二进制数依然保持不变，但Rust理解我们想遵循ASCII的惯例，于是打印出 `a` 只要我们要求。

ASCII码0-31都是不可打印字符（你会在里面找到铃声、回车和上方换行符），127也是控制字符。ASCII代码32-126均可打印。 `is_control()` 因此，只需查看代码的范围。<sup><a href="#fn4" id="fnref4">[4]</a></sup>

这是一个非常实用的程序，我们再多试试。它展示了各种按键如何转化为我们所读的字符。大多数普通键可以直接转化为它们所代表的字符。但试着看看会发生什么

- 当你按下方向键时，
- 或 `Escape`,
- 或 `Page Up`, `Page Down` , `Home`, `End` (`fn` 以及MacBook上的箭头上下左右），
- 或 `Backspace`, `Delete`, `Enter`.
- 确保你也尝试按键组合 `Ctrl`，比如 `Ctrl-A`, `Ctrl-B`，等等。

你会注意到一些有趣的事情：

- 压制 `Escape` 发送单人 `27` 字节作为输入。
- 方向键， `Page Up`, `Page Down`, `Home`， 和 `End` 所有输入3或4字节到终端： `27` （与Escape键相同）， `[`然后是一两个其他角色。这被称为 **逃脱序列**，键即以此命名。
- `Backspace` 是字节 `127`.
- `Enter` 是字节 `13` .如果你在ASCII表中查找，会发现这是一个回车字符，我们之前用它作为 `'\r'` - 而不是你所想的，是新行， `'\n'`
- 特殊字符如德语变音符号也会产生多个字节。
- `Ctrl-A` 是 `1`, `Ctrl-B` 是 `2`, `Ctrl-C` 是...... `3` 而且它并没有像你预期的那样终止程序（Raw模式会对我们关闭这个功能）。以及其他 `Ctrl` 密钥组合似乎将字母A-Z对应到代码1-26。  
  仔细观察字节表示，你会发现ASCII的设计使你可以从一个字符直接映射到其小写字母，或者 `CTRL` 而这个字符只需操作几个字节即可完成。例如， `CTRL` 似乎把上面的三个位设为0。

## 错误处理

让我们以改进处理错误的方式来结束本章（比如：以前我们没处理过错误，但现在想处理）。这也是我会兑现承诺，偶尔解释的地方 `unwrap()` 我们一路上见过面。

首先，简单说点理论。在许多其他编程语言中，如果函数遇到错误，它不会返回，而是“抛出一个异常”，你可以在更高位置捕捉到并尝试恢复。这意味着函数实际上有两种方式可以将控制权返回给调用者：要么返回一个值，要么抛出异常，而这两种方式都需要不同的代码来处理，使得推理代码变得更困难。

在 Rust 中，你可以像返回其他返回值一样返回函数错误，从而更简洁地处理例外情况。为了简洁地实现这一点，Rust 提出了一个概念 <a href="https://doc.rust-lang.org/book/ch09-02-recoverable-errors-with-result.html" class="external-link" target="_blank">结果</a>，该格式有两个变体： `Ok` 以及 `Err`.这里有一个比喻，我们将用一段时间：

假设我欠你钱，你向我要求这笔钱。那我要么给你钱，要么我告诉你不行，我什么都没给你。这是我想告诉你的，“请求欠款”是一个函数，要么是退还一些钱，要么会抛出异常。注意，如果我没有钱，退还0欧元，如果我真的欠你钱，那就无效了，所以这里破例更接近“拿回欠款”的心理模式。

现在，我不给你钱，而是还给你一个盒子，无论我有钱还是没钱，我都会这样做。如果我有你的钱，那是一个粉色盒子，上面贴着标签 `Ok`.包裹在里面，你会发现你需要的钱。否则就是黑盒子，上面有标签 `Err`.那个盒子里有我为什么没给你任何东西的信息（我没钱，或者说 *你* 欠款 *我* 钱？！）

现在——怎么办 `.unwrap()` 用比喻来说，做的就是假设盒子是粉色的，然后把结果拿出来。如果盒子是黑色的，程序就会当场停止。

我们想要的行为是尽可能向上传递错误——令 *他们* 去处理上面那个黑匣子！——在最高层面，当除了我们自己没有人可以传递错误时，我们最终会终止这个计划。
