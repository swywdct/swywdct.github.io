## 高亮架构的设置

在深入突出之前，我们必须先讨论建筑学的考虑因素。让我先向你介绍问题空间，然后在教程的范围内解释我打算如何解决它。解决这个问题的方法有很多，你可以自己选择。

目前，我们简单的高亮功能就在 `Line`.这适用于大多数情况，但并非所有情况：有些评论可以跨越多行，比如这样：

``` rust
// This is a regular comment
/* 
This is a multi line comment
It goes on
And on
until here: */
```

为了正确突出这一点，我们需要有关于前一行的信息：如果它结尾为多行注释，当前行则从多行注释开始。

这在更新高亮信息时也带来了一些挑战：我们不能推断任何变化只影响该行的高亮——如果你开始多行评论，后续所有行都需要重新高亮。

此时，重要的是：高亮逻辑需要从中移出 `Line`.  
我们可以把它移到周围的结构里， `Buffer`然后从那里引导高亮，或许把高亮信息本身（以我的代码为例：注释）存储在 `Line`.

事实上，这正是我在早期版本中解决这个问题的方法。 `hecto`:

- 我定义了一个枚举，用来确定高亮类型
- 每人 `Line` 包含这些高亮类型的向量，每个字符在行中
- 周围结构体负责确保该向量始终与文档本身同步，这与我们目前确保片段向量与文档内字符串同步的方式非常相似 `Line`.

这是一个有效且可行的方法。

在这一迭代中 `hecto`我将走一条更接近其他文本编辑器实现方式的路线：我将创建一个专门关注高亮的独立结构。它会为每行存储高亮信息，然后传递给 `Line` 在构建最终的注释字符串时。我的实现会更注重简洁，而不是性能，但有了这种设置，之后做性能调整会更容易——比如只重新高亮文档直到当前可见屏幕的边缘。

随你挑！不过我认为这两种方法都不会比另一种更容易实施。

``` rust
    // Create a new HashMap
    let mut scores = HashMap::new();

    // Insert some key-value pairs
    scores.insertfrom("Alice"), 50;
    scores.insertfrom("Bob"), 40;
    scores.insertfrom("Carol"), 100;

    // Access a value by key
    let alice_score = scores.get("Alice");
    println!("Alice's score: {:?}", alice_score);

    println!("All scores:");
    for (key, value) in &scores {
        println!("{}: {}", key, value);
    }

    // Remove a key-value pair
    scores.remove("Carol");
    println!("After removing Carol, scores are:");
    for (key, value) in &scores {
        println!("{}: {}", key, value);
    }
}
```

<a href="https://github.com/pflenker/hecto-tutorial/commit/43fa7f36540e6971c3a247d74c57a2ca49d4cccb#diff-7b05039899d019f967babe4ee53ea3cd9cbf37a682ee09b16bf3d1ba35d64d96R13" class="external-link" target="_blank">以下是我解决这个问题的方法。</a>

在我的代码中，我想展示一个常见场景：我们有一个通用的 *事情*，一个语法高亮器，我们可以有具体的语法高亮器：一个用于Rust，一个用于TOML，应有尽有。在Rust中，这个问题通常通过特征来解决，果然：我实现了一个特征，叫做 `SyntaxHighlighter` 针对我代码中目前可用的两个高亮。因此，将这种方式扩展到第三支高亮器应该很容易。

但我对引用语法高亮的结构体成员的定义看起来很奇怪，看起来类似：

``` rust
pub struct Highlighter {
    syntax_highlighter: Box<dyn SyntaxHighlighter>,
    //...
}
```

在 Rust 中，字段类型必须固定且在编译时已知。呼唤 `highlighter.syntax_highlighter.highlight();` 需要 Rust 知道 的定义在哪里 `highlight()` 是存储的。只有当它知道结构体成员的类型时，它才能做到这一点，仅知道在哪里（=指向它的指针）是不够的。这里的理念是，抽象不应该产生运行时间开销，因此访问 `highlight()` 运行时无需额外计算来确定函数定义的位置。

但在这种情况下，我们需要有运行时开销，这就是 `Box` 进来了。我们之前见过Boxes，当时我们讨论了恐慌钩，现在我们自己也在用一个。通过使用 `Box<dyn SyntaxHighlighter>` 我们告诉 Rust 我们想要动态调度，也就是说，在运行时而不是编译时，我们需要确定给定特征的特征方法分布在哪里。结构体， `Highlighter`，存储的不仅仅是简单的指针。 `syntax_highlighter` - 存储一个指向目标本身的智能指针 *以及* 一个查找表，使Rust在运行时能够为混凝土找到正确的方法 `SyntaxHighlighter`.

与盒子互动和与物体本身互动没太大区别，只是制造盒子看起来稍微不同：

``` rust
fn create_syntax_highlighter(file_type: FileType) -> Box<dyn SyntaxHighlighter> {
    match file_type {
        FileType::Rust => Box::<RustSyntaxHighlighter>::default(),
        //...
    }
}
```

## 第39次任务：色彩斑斓的斑点

我们现在需要超越逐字解析前几步：我们要正确突出角色。为此，我们需要突出显示两者之间的所有内容 `'`S——我们不会进一步核查两者之间的所有内容是否 `'` 实际上 *是* 一个有效的角色，所以我们会重点介绍 `'definitelynotachar'` 尽管那不是角色。  
但事情并不像寻找成交那么简单 `'` 然后试图解析中间的一切，因为 `'` 本身是 *还有* 一个角色！为了表现这一点，我们需要告诉Rust，这件事并不是为了结束角色，我们通过在前面加上一个 `\` （这被称为逃脱角色——不要和我们之前遇到的另一种逃脱角色混淆）——所以这个角色 `'` 表示为 `'\''` .对于字符，我们可以假设 `\` 最多发生一次：紧接开头引言之后。

我们已经无法逐字解析，因为 `’a'` 以三个词的形式出现 `'`, `a` 以及 `'`.我们将调整策略：不再将字传递给每个解析函数，而是传递剩余字符串的引用。而且没有回去 `true` 或 `false`，我们将返回一个 `Annotation` - 并根据注释的终点索引推进周围环的迭代子。

换句话说：在解析过程中 `This is a char: ‘a’!`，我们将对该字符串中的单词进行迭代，直到到达第一个 `’`.然后我们将剩余字符串传递给解析函数： ` ‘a’!` .该函数返回一个注释，尾号为第二个字节的末尾索引 `'`.我们需要调整外环，直到该字节索引后再进行高亮。

我们用琥珀色来表示角色。

这是作业内容：

- 通过突出开头和结尾之间的所有内容来突出角色 `'`.
- 忽略任何逃脱的 `'` （即 `\’`)

## 作业41：彩色单行评论

让我们确保（单行）评论被清晰地标示出来。正如你现在所知，Rust有两种单行评论：普通评论以两条斜杠开头（`//`).这两条斜线之后直到行尾的所有内容都被视为评论。这些内容用于记录每个函数的内部结构，面向那些在你函数中推理代码的开发者。  
还有文档注释，开头有三条斜杠： `///`.这些文件可以出现在自动生成的文档中，因此主要面向调用函数并与返回值互动的人。按照惯例，三斜杠注释描述该注释之后的代码。如果你想记录文件所在的物品（比如箱子本身），可以用 `//!`.这里有一个使用了三种评论类型的示例：

``` rust
//! # Comments
//! 
//! A file which showcases all kinds of single-line comments in Rust.

/// A sample struct which is documented with this comment.
struct hecto;

impl hecto {
   /// Outputs Hello, World. This comment shows up in the auto-generated documentation around say_hello.
   fn say_hello() {
    println!("Hello, World!"); // using a macro here. This comment doesn't show up in the auto-generated documentation.
   }
}
```

让我们重点介绍这些。在我看来（但你可能不同意），用不同方式突出评论类型并不太合理，因此简单的实现只需检查 `//` 为了察觉到评论然后继续前进。

现在就实现这个功能，并用更深的绿色高亮评论。

## 作业42：多彩多行评论

我们已经到了这一章的后半段，需要处理一个更复杂的案例：多行评论。  
多行评论以 `/*`，这可以发生在直线内的任意位置。该行的其余部分被视为注释，之后所有行都被视为，直到包含 `*/` - 然后，这行的其余部分会定期被高亮显示。  
一个需要注意的是，存在嵌套多行注释，例如：

``` rust
/* This is an outer ML comment.
    /* This is an inner one! 
        Once this comment ends... */
 ...the text outside is still part of the comment.
*/
```

这时我们只一次标注一行的策略已不够：高亮后续句子需要之前的句子也被高亮，而且我们需要延续了我们见过的多少悬而未决的ML评论开场白。

我们的策略将是这样的：

- 一旦我们遇到 `/*`，我们继续迭代字符串的其余部分。
- 我们会记录像这样未关闭的机器学习评论：
  - 我们从1开始（第一次） `/*` 我们遇到了）
  - 对于后续的每个 `/*` 我们看到了，我们加了1
  - 对于后续的每个 `*/` 我们看到了，我们减去一
- 一旦达到0，我们会将给定部分作为注释进行注释，然后进行常规的高亮标记。
- 如果我们在到达最终之前就完成了弦的终点 `*/`，我们将剩余的数字作为结构体的成员存储。
- 在开始常规高亮下一行之前，我们先检查是否处于机器学习评论中，并用上述算法继续注释当前行。
